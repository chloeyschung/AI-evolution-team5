# IOS-010: iOS AI 요약 폴링 타이밍 수정 (AI Summary Polling Fix)

**Status**: Draft | **Created**: 2026-06-04 | **Author**: suyoung
**F-xxx Mapping**: F-005 (AI Summary - Real-time Generation) | **Phase**: Phase 3 - Bug Fix | **Priority**: Critical

---

## 1. Overview

**Problem**: 웹 브라우저에서는 AI 요약이 정상 동작하지만, iOS 앱에서는 항상 "요약을 준비 중입니다." 상태로 고착된다.

**Root Cause 분석 (2026-06-04)**:

iOS 앱의 요약 폴링 윈도우(12초)가 Modal GPU 서버의 처리 시간(30~55초)보다 훨씬 짧다.  
웹은 페이지를 열어두는 동안 계속 기다리지만, iOS는 12초 후 포기한다.

**Solution**: iOS 폴링 횟수·간격 확대 + Share Extension race condition 수정

---

## 2. 문제 진단 상세

### 2-1. iOS 폴링 윈도우 부족 (버그 1 — 핵심)

**파일**: `Briefly/Briefly/Views/ItemDetailView.swift:370`

```swift
// 현재 코드 (문제)
for attempt in 0..<5 {
    if attempt > 0 {
        try? await Task.sleep(nanoseconds: 3_000_000_000)  // 3초 간격
    }
    // GET /api/v1/content/{id} 폴링
}
```

| 항목 | 시간 |
|------|------|
| iOS 최대 대기 시간 | **12초** (0 + 3 + 3 + 3 + 3) |
| Modal GPU cold-start | **30~50초** |
| 백엔드 summarizer 총 타임아웃 (PR #31) | **55초** |

iOS가 폴링을 포기한 뒤 20~43초 후에 summary가 DB에 저장된다.  
백엔드는 `200 OK + summary: null`을 반환하므로 iOS는 에러 없이 조용히 실패한다.

**수정 방향**: 폴링을 `15회 × 5초 = 총 70초`로 변경하여 55초 백엔드 타임아웃을 완전히 커버.

---

### 2-2. Share Extension Race Condition (버그 2)

**파일**: `BrieflyShareExtension/ShareViewController.swift:69`

```swift
private func saveAndShow(url: URL) {
    var item = SavedItem(url: url, title: nil)
    StorageService.shared.appendToInbox(item)   // serverContentId=nil로 먼저 저장

    Task {
        let result = try await BrieflyAPI.shared.share(url: url, token: token)
        item.serverContentId = result.id         // API 응답 후 비동기 업데이트
        StorageService.shared.updateItemById(item)
    }

    // ← Task 완료를 기다리지 않고 즉시 확인 UI 표시
    DispatchQueue.main.async { self.showConfirmation(success: true, savedURL: url) }
}
```

사용자가 확인 UI에서 "지금 읽기"를 API 응답 전에 누르면 `serverContentId=nil` 상태로 `ItemDetailView`가 열린다.  
`ItemDetailView:365`의 `guard let contentId = currentItem.serverContentId` 조건에서 막혀 폴링 자체가 시작되지 않는다.

**수정 방향**: Task 완료 후 또는 `serverContentId`를 전달받은 뒤 확인 UI를 표시하도록 흐름 조정.

---

### 2-3. 백엔드 app.py 타임아웃 설정 불일치 (버그 3 — 백엔드)

**파일**: `src/api/app.py:122`

```python
summarizer = Summarizer(
    ...
    timeout=120.0 if modal_tokens_ok else None,   # per-attempt 120초 설정
    # total_timeout은 명시하지 않아 DEFAULT_TOTAL_TIMEOUT=55s 사용됨
)
```

PR #31에서 `Summarizer`에 2단계 타임아웃 패턴을 도입했으나, `app.py`는 `timeout=120.0`만 넘기고 `total_timeout`은 여전히 기본값(`55.0`)을 사용한다.  
`120s > 55s`이므로 per-attempt 타임아웃(120초)은 total_timeout(55초)에 의해 항상 먼저 캔슬되어 무의미하다.  
Modal 서버에서 cold-start가 50초를 넘으면 55초 total_timeout에 걸려 요약이 실패한다.

**수정 방향**: `app.py`에서 `total_timeout=120.0`도 함께 명시.

## 5. AI 모델 (Gemma) 상태 평가

Modal 서버의 Gemma 모델 자체는 정상 작동한다고 판단한다:

| 항목 | 상태 |
|------|------|
| OpenAI-compatible API 포맷 | ✅ `summarizer.py`가 URL 기반 auto-detect로 "openai" provider 사용 |
| 인증 헤더 (Modal-Key/Modal-Secret) | ✅ `app.py:105-107`에서 extra_headers 설정 |
| 응답 파싱 (`choices[0].message.content`) | ✅ `_extract_summary()` "openai" 브랜치 처리 |
| Thinking 토큰 차단 | ✅ `enable_thinking: False` (vLLM reasoning model 호환 설정) |
| Modal 서버 uptime | ⚠️ `abraxaspark91` 계정 종속 — 서버가 꺼지면 요약 불가 |

**결론**: Gemma 모델 호출 코드 자체에는 문제 없음.  
단, cold-start(30~50초)와 현재 타임아웃 설정(total 55초)의 마진이 좁아 간헐적 실패 가능.  
버그 3 수정 시 이 마진이 65초로 개선되어 신뢰성 향상.
