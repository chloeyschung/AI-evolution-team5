# IOS-001: iOS ↔ 백엔드 API 연동 (인증 + Share Extension)

**Status**: Implemented | **Created**: 2026-04-30 | **Last Updated**: 2026-04-30 | **Author**: suyoung
**F-xxx Mapping**: iOS App API Integration | **Phase**: Phase 2 - Backend Integration | **Priority**: Critical

---

## 1. Overview

**Problem**: iOS 앱이 로컬 UserDefaults만 사용해 데이터를 저장하고 있어, 다기기 동기화와 서버 측 AI 요약이 불가능하다.

**Solution**: 이메일/비밀번호 인증으로 로그인하고, Share Extension에서 URL을 `POST /api/v1/share`로 백엔드에 전송한다.

**Goals**:
- 로그인한 사용자는 URL 저장 시 백엔드에도 자동 전송
- 로그인 없이도 로컬 저장은 정상 동작 (오프라인 지원 유지)

**Non-Goals**:
- Apple Sign-In 지원 (@ARCH-017)
- Google OAuth iOS 구현 (향후 IOS-002로 분리 예정)
- 다기기 동기화 전체 구현 (현재는 쓰기만)
- 기존 로컬 항목의 서버 마이그레이션

## 3. User Story / Behavior

**로그인 흐름:**
1. Account 탭 → 이메일 + 비밀번호 입력 → 로그인 버튼
2. `POST /api/v1/auth/login` 호출
3. 성공: `access_token`, `refresh_token`을 App Group UserDefaults에 저장
4. Account 탭에 로그인 상태(이메일) 표시

**Share Extension URL 저장 흐름:**
1. Safari 등에서 공유 → Share Extension 실행
2. 로컬 UserDefaults inbox에 저장 (기존 동작 유지)
3. `AuthTokenStore.shared.accessToken`이 존재하면 `POST /api/v1/share` 호출
4. API 성공/실패 관계없이 확인 UI 표시

## 4. Data Models

### Existing Used

- `SavedItem` — 로컬 캐시 모델 (URL, savedAt, fetchStatus 등)
- `StorageService` — App Group UserDefaults inbox/drain 패턴

### New (iOS-only)

**AuthTokenStore** (App Group UserDefaults, key prefix `briefly.auth.*`):

| Key | 값 |
|-----|-----|
| `briefly.auth.access_token` | JWT access token (String) |
| `briefly.auth.refresh_token` | Refresh token (String) |
| `briefly.auth.user_id` | User ID (Int) |
| `briefly.auth.display_name` | 표시 이름 (String?) |
| `briefly.auth.email` | 이메일 (String?) |

## 5. API Design

### 사용하는 백엔드 엔드포인트

| Endpoint | Method | 설명 |
|----------|--------|------|
| `/api/v1/auth/login` | POST | 이메일/비밀번호 로그인 |
| `/api/v1/auth/refresh` | POST | 토큰 갱신 |
| `/api/v1/auth/logout` | POST | 로그아웃 (토큰 무효화) |
| `/api/v1/share` | POST | URL 공유 저장 |
| `/api/v1/user/device-token` | POST | APNs 푸시 토큰 등록 (향후) |

### 로그인 Request/Response

**POST /api/v1/auth/login**
```json
// Request
{ "email": "user@example.com", "password": "..." }

// Response 200
{
  "access_token": "eyJ...",
  "refresh_token": "abc...",
  "expires_at": "2026-04-30T10:00:00Z",
  "user_id": 42,
  "email": "user@example.com"
}
```

**POST /api/v1/share**
```json
// Request (Authorization: Bearer {access_token})
{ "content": "https://example.com/article", "platform": "example.com" }

// Response 201
{
  "id": 1,
  "url": "https://example.com/article",
  "title": null,
  "platform": "example.com",
  "created_at": "2026-04-30T09:00:00Z"
}
```

## 6. Implementation

### Files

- `Briefly/Services/AuthTokenStore.swift` — App Group UserDefaults 토큰 저장소 (양 타겟)
- `Briefly/Services/BrieflyAPI.swift` — URLSession API 클라이언트 (양 타겟)
- `Briefly/Views/AccountView.swift` — 이메일 로그인 UI + 로그인 상태 표시
- `BrieflyShareExtension/ShareViewController.swift` — URL 저장 후 API 전송 추가

### Xcode Target Membership 필요

`AuthTokenStore.swift`와 `BrieflyAPI.swift`는 두 타겟 모두에 추가 필요:
- ☑ Briefly (메인 앱)
- ☑ BrieflyShareExtension

### Key Logic

1. 앱 시작 시 `AuthTokenStore.shared.isLoggedIn` 확인
2. 로그인: `BrieflyAPI.shared.loginWithEmail()` → 성공 시 `AuthTokenStore.shared.save()` 호출
3. Share Extension: URL 저장 후 토큰 있으면 `BrieflyAPI.shared.share()` 호출
4. API 오류는 `print()`만 하고 묵살 (로컬 저장이 우선)

### Dependencies

**Requires** (satisfied by):
- `AUTH-001` - JWT 토큰 기반 세션 관리 (백엔드)
- `AUTH-005` - 이메일/비밀번호 인증 (백엔드)
- `ING-001` - Share Handler (`POST /api/v1/share`)

**Provides** (for future):
- `IOS-002` - Google OAuth iOS 구현 (GoogleSignIn SDK)
- `IOS-003` - 다기기 읽기 동기화 (`GET /api/v1/content`)

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| 비로그인 상태에서 Share | 로컬만 저장, API 호출 없음 |
| API 호출 실패 (네트워크 없음) | `try?`로 묵살, 로컬 저장 완료 후 UI 표시 |
| 토큰 만료 | 현재: 재로그인 필요. 향후: 자동 refresh 구현 |
| 이메일 미인증 상태 로그인 시도 | 백엔드 403 반환 → 에러 메시지 표시 |
| 잘못된 이메일/비밀번호 | 백엔드 401 반환 → "이메일 또는 비밀번호가 올바르지 않습니다" 표시 |
