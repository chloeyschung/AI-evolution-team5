# IOS-007: 앱 저장 콘텐츠 Auto-Tag 연동 (iOS Auto-Tag Integration)

**Status**: Draft | **Created**: 2026-05-29 | **Author**: suyoung
**F-xxx Mapping**: iOS Auto-Tag Pipeline | **Phase**: Phase 4 - UX | **Priority**: Medium

---

## 1. Overview

**Problem**: IOS-006의 홈 탭 주제별 섹션은 서버의 동적 클러스터(IOS-008)에 의존하며, 클러스터 품질은 각 콘텐츠에 얼마나 풍부한 **AI 요약 · 키워드 · 카테고리** 데이터가 있는지에 달려 있다. Share Extension으로 저장된 콘텐츠가 서버에서 AI 파이프라인(요약·자동 태깅)을 완료한 후, iOS 앱이 그 결과를 로컬에 반영하는 흐름이 없다. 또한 `SavedItem` 로컬 모델에 auto-tag 필드가 없어 폴백 모드(고정 카테고리)에서도 로컬 아이템이 주제별 섹션에서 누락된다.

**Solution**: 
1. Share Extension → `POST /api/v1/share` 후 서버가 백그라운드로 AI 요약·자동 태깅 파이프라인을 실행한다 (서버 측 이미 구현).
2. iOS 앱은 포그라운드 진입 시 `GET /api/v1/content`로 서버의 최신 **summary · auto-tag** 결과를 fetch하여 로컬 `SavedItem`에 병합한다.
3. `SavedItem` 로컬 모델에 `autoTagCategory`, `autoTagKeywordsEn`, `autoTagKeywordsOriginal` 필드를 추가하고, `summary`도 서버 최신값으로 동기화하여 IOS-008 동적 클러스터링 입력 품질을 높인다.

**Goals**:
- Share Extension으로 저장한 콘텐츠의 AI 요약·자동 태깅 결과가 앱 포그라운드 진입 시 로컬에 반영된다
- 로컬 `SavedItem`의 summary · auto-tag 필드가 서버 최신 상태로 유지되어 IOS-008 클러스터링 입력 품질이 보장된다
- 앱 재시작 후 오프라인 상태에서도 마지막 sync된 데이터로 주제별 섹션을 표시한다

**Non-Goals**:
- iOS 기기에서 직접 LLM API를 호출하는 온디바이스 auto-tag — 서버 의존 방식 유지
- 실시간 태깅 결과 푸시(Push Notification) 수신 — 향후 개선
- auto-tag 결과를 사용자가 직접 수정하는 UI

## 3. User Story / Behavior

**As a** Briefly 앱 사용자,
**I want to** Share Extension으로 저장한 링크가 서버에서 자동 분류된 후 홈 탭 주제별 섹션에 나타나길 원한다,
**so that** 앱으로 저장한 콘텐츠도 웹과 동일하게 주제별로 묶어볼 수 있다.

### Key Behaviors

```
[Share Extension으로 링크 저장]
  → POST /api/v1/share → 서버에 콘텐츠 생성
  → 서버 백그라운드: auto_tagger.tag() 실행 → DB에 auto_tag_* 필드 저장
     (apply-auto-tag 파이프라인, 수 초~수십 초 소요)

[앱 포그라운드 진입 (IOS-006 HomeViewModel.load())]
  → GET /api/v1/content fetch (기존 IOS-006 흐름)
  → 응답의 각 ServerContent를 로컬 SavedItem과 serverContentId로 매칭
  → 병합 필드:
    - auto_tag_category, auto_tag_keywords_en/original (서버 값이 있을 때만 덮어씀)
    - summary (로컬 summaryStatus != .done인 경우에만 업데이트)
  → StorageService.mergeServerData() 로 로컬 저장
  → HomeViewModel 주제별 섹션에 반영 (IOS-008 동적 클러스터 또는 auto_tag 폴백)

[오프라인 앱 재시작]
  → StorageService에서 로컬 SavedItem 로드
  → autoTagCategory 필드가 있으면 주제별 섹션에 포함
  → 서버 fetch 없이도 마지막 auto-tag 결과 표시
```

## 6. Implementation

### Files 변경 목록

| 파일 | 변경 내용 |
|------|-----------|
| `Briefly/Models/SavedItem.swift` | `autoTagCategory`, `autoTagKeywordsEn`, `autoTagKeywordsOriginal` 필드 추가 |
| `Briefly/Services/StorageService.swift` | `mergeServerData(_:)` 메서드 추가 — summary · auto-tag 필드 병합 |
| `Briefly/ViewModels/HomeViewModel.swift` | 서버 fetch 후 `mergeServerData()` 호출 |
| `Briefly/Models/HomeItem.swift` (IOS-006) | `autoTagCategory` 로직 업데이트 (폴백 모드에서 사용) |

### `mergeServerData` 핵심 로직

```swift
func mergeServerData(_ serverItems: [ServerContent]) {
    var items = loadAll()
    var updated = false
    for serverItem in serverItems {
        guard let idx = items.firstIndex(where: { $0.serverContentId == serverItem.id }) else { continue }
        // auto-tag 결과 병합 (서버 값이 있을 때만 덮어씀)
        if let category = serverItem.autoTagCategory {
            items[idx].autoTagCategory = category
            items[idx].autoTagKeywordsEn = serverItem.autoTagKeywordsEn
            items[idx].autoTagKeywordsOriginal = serverItem.autoTagKeywordsOriginal
        }
        // AI 요약 병합 (로컬에 아직 없는 경우만 — 폴링 완료 전 갭을 메움)
        if items[idx].summaryStatus != .done, let summary = serverItem.summary {
            items[idx].summary = summary
            items[idx].summaryStatus = .done
        }
        updated = true
    }
    if updated { saveAll(items) }
}
```

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| `apply-auto-tag` 미머지 (서버가 nil 반환) | `autoTagCategory == nil`이면 로컬 필드 덮어쓰지 않음 |
| `serverContentId` 없는 로컬 아이템 (비로그인 저장) | 매칭 불가 → 주제별 섹션 제외 (IOS-005 sync 후 재시도) |
| auto-tag가 아직 처리 중인 콘텐츠 | 서버 `auto_tag_status == "pending"` → category nil → 주제별 섹션 제외 |
| 기존 `SavedItem` JSON (신규 필드 없음) | optional + 기본값으로 디코딩 성공 |
| 서버 fetch 실패 | 기존 로컬 auto-tag 데이터 유지 |
