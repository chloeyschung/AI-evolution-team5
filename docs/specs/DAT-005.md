# DAT-005: User Timezone Storage for Mobile Context

**Status**: Design | **Created**: 2026-04-18 | **Author**: codex
**F-xxx Mapping**: N/A (Profile/Data Contract Hardening) | **Phase**: Phase 1 - MVP (Mobile Focus) | **Priority**: High

---

## 1. Overview

**Problem**: User profile currently stores timestamps only, without explicit timezone preference/context. This prevents reliable local-time rendering and scheduling semantics for mobile users.

**Solution**: Add `timezone` (IANA TZ string, e.g. `Asia/Seoul`) to `UserProfile`, expose it in profile read/update APIs, and default to `UTC` for backward compatibility.

**Goals**: Persist timezone per user, allow profile-driven timezone updates, keep UTC timestamp storage unchanged.

**Non-Goals**: Converting all historical timestamp outputs to local time server-side, DST-aware notification scheduler redesign, geolocation-based timezone auto-detection.

---

## 3. User Story / Behavior

As a mobile user, I want my preferred timezone stored in my profile so app features can interpret dates and times in my local context.

### Key Behaviors

- Profile GET returns timezone
- Profile PATCH can update timezone independently of other profile fields
- Unset/legacy profile rows resolve to `UTC`

---

## 4. Data Models

### Schema Changes

**UserProfile**:
- `timezone: String(64), nullable=False, default='UTC'`

### Existing Used

- `UserProfileRepository.get_or_create_profile()`
- `UserProfileRepository.update_profile()`

---

## 5. API Design

| Endpoint | Method | Change |
|----------|--------|--------|
| `/api/v1/profile` | GET | Add `timezone` in response |
| `/api/v1/profile` | PATCH | Accept optional `timezone` update |

### Example

```json
{
  "id": 1,
  "display_name": "Younghwan",
  "avatar_url": null,
  "bio": null,
  "timezone": "Asia/Seoul",
  "created_at": "2026-04-18T01:10:00Z",
  "updated_at": "2026-04-18T12:42:05Z"
}
```

## 6. Implementation

### Files

- `src/data/models.py` - add `UserProfile.timezone`
- `src/data/repository.py` - initialize and update timezone in profile repository
- `src/api/schemas.py` - profile request/response timezone fields
- `src/api/routers/user.py` - timezone wiring in profile GET/PATCH
- `tests/data/test_user_profile.py` - repository timezone behavior tests
- `tests/api/test_account_user_datetime.py` - API-level timezone round-trip tests

### Key Logic

1. Ensure model default `UTC` for new profile rows.
2. Pass timezone through repository update path when provided.
3. Return timezone from profile endpoints consistently.

### Dependencies

**Requires**:
- DAT-002 (profile model and profile endpoints)

**Provides**:
- timezone context for reminder/scheduling and iOS rendering layers

---

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Legacy row with null timezone | fallback to `UTC` in response path |
| PATCH without timezone | keep existing timezone unchanged |
| Empty string timezone | reject by schema validation |
