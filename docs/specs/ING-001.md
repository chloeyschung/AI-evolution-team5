# ING-001: Mobile Share Sheet Integration

## User Story
As a time-poor professional, I want to save content to Briefly via my device's native share sheet, so that I can capture information instantly with a single tap without context switching.

## Implementation Contract

### Level 1: Natural Language
Mobile Share Sheet Integration provides the primary entry point for content ingestion on mobile devices. It intercepts share actions from the OS-level share sheet (iOS/Android) and processes various content types (URLs, text, images, app-specific deep links) into a standardized format for downstream processing.

The service must:
- Accept heterogeneous share data from the OS share sheet
- Auto-detect content type without user intervention
- Route to appropriate processing pipeline based on detected type
- Return structured metadata for further AI processing

### Level 2: Logic Flow (Pseudocode)
```
INPUT: share_payload (dict/object from OS share sheet)
PROCESS:
  1. Validate payload contains required content field.
  2. Detect content type:
     - If explicit type hint provided, use it.
     - Otherwise auto-detect based on content format:
       a. HTTP/HTTPS URL pattern → URL type
       b. Custom scheme (e.g., app://) → Deep link type
       c. Image data (base64 or image URL) → Image type
       d. Fallback → Plain text type
  3. Route to appropriate processor based on detected type.
  4. Extract and normalize metadata from share payload.
  5. Return standardized content metadata object.
OUTPUT: content_metadata (structured object with platform, type, source)
ERROR: ShareDataError (invalid payload, empty content, unsupported type)
```

## Verification Contract

### Level 1: Scenarios (Natural Language)
- **Success:** A user shares a news article URL via iOS share sheet; system detects URL type and extracts metadata.
- **Success:** A user shares plain text from a messaging app; system treats it as text content.
- **Success:** A user shares from a social media app (e.g., WhatsApp); system detects deep link and identifies source.
- **Success:** A user shares an image from gallery; system recognizes image data format.
- **Failure:** Share payload is missing required content field; system raises validation error.
- **Edge:** Shared text contains an embedded URL; system extracts URL for further processing.

### Level 2: Given-When-Then
```
GIVEN a user shares a YouTube URL via iOS share sheet
WHEN process_share is called with the share payload
THEN the content type is detected as URL and source_platform is identified

GIVEN a user shares plain text without any URL
WHEN process_share is called
THEN the content type is detected as plain text and source_platform is "clipboard"

GIVEN a user shares from WhatsApp with a deep link scheme
WHEN process_share is called
THEN the content type is detected as deep link and source_platform is "whatsapp"

GIVEN a share payload with empty content field
WHEN process_share is called
THEN a ShareDataError is raised

GIVEN a share payload with text containing an embedded URL
WHEN process_share is called
THEN the URL is extracted and included in the output
```

## Implementation Notes
- The OS share sheet integration (iOS/Android native code) is implemented in the React Native app layer; this spec covers the backend processing service.
- Content type detection should be resilient: if auto-detection fails, default to plain text rather than failing.
- Preserve all available metadata from the share payload for downstream services to use.
- Use async/await pattern for consistency with other ingestion services.
