## 1. Overview

**Problem**: Application requires comprehensive security measures to protect user data, prevent common web vulnerabilities, and ensure multi-user data isolation.

**Solution**: Multi-layered security implementation covering authentication, data encryption, input validation, rate limiting, and access control.

**Goals**: OWASP Top 10 compliance, secure multi-tenant architecture, data protection at rest and in transit

**Non-Goals**: 2FA, device management, audit logging (future enhancements)

## 3. User Story / Behavior

As a Briefly user, I want my data to be secure and isolated from other users, so that I can trust the platform with my saved content and personal information.

### Key Behaviors

- JWT access tokens hashed (SHA-256) before database storage
- OAuth tokens encrypted (Fernet) at rest
- Rate limiting prevents abuse (10/min for /share, 30/min for /ingest)
- SSRF protection blocks private IP access
- All queries filtered by user_id (data isolation)
- Thread-safe components prevent race conditions

## 5. API Design

### Security Headers

All responses include:
- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: DENY`
- `X-XSS-Protection: 1; mode=block`

### Authentication Endpoints

| Endpoint | Auth Required | Rate Limit |
|----------|---------------|------------|
| `POST /auth/google` | No | 5/minute |
| `POST /auth/refresh` | Yes (refresh token) | 10/minute |
| `POST /auth/logout` | Yes | 10/minute |
| `DELETE /auth/account` | Yes | 5/minute |

### Protected Endpoints

All content, swipe, achievement, and reminder endpoints require:
```
Authorization: Bearer <access_token>
```

Token validated on every request via `Depends(get_current_user)`.

### Error Responses

**401 Unauthorized**:
```json
{
  "detail": "Could not validate credentials"
}
```

**429 Too Many Requests**:
```json
{
  "detail": "Rate limit exceeded"
}
```

### Files Created

- `src/utils/token_encryption.py` - Fernet OAuth encryption
- `src/middleware/rate_limiter.py` - Token bucket rate limiting

### Files Modified

- `src/data/auth_repository.py` - Hashed token storage
- `src/data/models.py` - Token encryption methods, user_id fields
- `src/data/repository.py` - User-filtered queries
- `src/ingestion/extractor.py` - SSRF protection
- `src/api/routes.py` - Auth dependencies, rate limit decorators
- `src/config.py` - Secret key validation
- `src/utils/http_client.py` - Thread-safe singleton

### Security Measures

1. **JWT Token Security**:
   - SHA-256 hashing before storage
   - Secret key validation (min 32 chars)
   - HMAC verification on every request
   - Plaintext JWT only sent to client, never stored

2. **OAuth Token Encryption**:
   - Fernet symmetric encryption
   - Key normalization (SHA-256 hash of key)
   - Getter methods for encrypted access
   - Centralized decryption logic

3. **Rate Limiting**:
   - Per-IP token bucket implementation
   - Configurable limits per endpoint
   - Graceful 429 responses
   - `/share`: 10/min, `/ingest`: 30/min

4. **SSRF Protection**:
   - DNS rebinding prevention (resolve and validate IP)
   - Private IP block (10.x, 172.16-31.x, 192.168.x)
   - URL validation (strict parsing)
   - Block localhost and link-local addresses

5. **Multi-User Data Isolation**:
   - user_id field on all user tables
   - Repository methods require user_id parameter
   - API routes use Depends(get_current_user)
   - Unique constraints: (user_id, tag) on InterestTag

6. **Thread Safety**:
   - HttpClientPool: threading.RLock() on singleton
   - Token encryption: module-level lazy init
   - Safe concurrent access patterns

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Invalid JWT signature | 401 Unauthorized |
| Expired access token | 401, client must refresh |
| Invalid refresh token | 401, re-authentication required |
| Rate limit exceeded | 429 with retry-after header |
| SSRF attempt detected | 400 Bad Request, log security event |
| Cross-user data access attempt | 404 (don't reveal existence) |
