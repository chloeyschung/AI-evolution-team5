# IOS-013: AI 요약 독립 타임아웃 및 수동 재시도 (AI Summary Isolation & Manual Retry)

**Status**: In Progress | **Created**: 2026-06-04 | **Updated**: 2026-06-04 | **Author**: suyoung  
**F-xxx Mapping**: F-005 (AI Summary - Real-time Generation) | **Phase**: Phase 3 - Bug Fix | **Priority**: Critical

---

## 1. Overview

**Problem**: 여러 링크를 연속 저장할 때 특정 항목(B)의 AI 요약이 실패/타임아웃되면, 이후 저장된 항목(C, D...)도 연쇄적으로 요약이 실행되지 않는다. 또한 타임아웃 후 재시도 방법이 없어 해당 항목은 "요약을 준비 중입니다." 상태로 영구 고착된다. 과거에 저장한 항목도 요약이 없으면 재시도할 방법이 없다.

**Solution**:
1. `FetchCoordinator` 순차 처리 → 병렬 처리 전환: B의 지연이 C, D에 영향을 주지 않도록
2. `SavedItem`에 `summaryStatus` 필드 추가: 타임아웃 시 실패 상태를 영구 저장 (다음 세션 자동 재폴링 방지)
3. `ItemDetailView` UI 단순화: `summary 없음 + 로딩 아님` → 무조건 "AI 요약 다시 시도" 버튼 표시 (타임아웃 항목, 과거 항목, serverContentId 없는 항목 모두 포함)

## 2. 문제 진단

### 2-1. FetchCoordinator 순차 처리 (1차 원인)

**파일**: `Briefly/Briefly/Services/FetchCoordinator.swift:29`

```swift
// 문제 코드
for item in pending {
    await fetchOne(item: item)   // 순차 실행 — B가 완료돼야 C 시작
}
```

`fetchOne` 1개당 최대 소요 시간:

| 단계 | 최대 소요 |
|------|-----------|
| OG 메타데이터 (URLSession) | ~10s |
| 본문 취득 URLSession + WKWebView 폴백 | ~20s |
| **항목 1개 최대** | **~30s** |

항목 4개 순차 처리 시 D의 rescan 호출이 최대 90초 지연된다.  
백엔드 total_timeout(120s)에 걸려 C, D의 요약이 누락될 수 있다.

### 2-2. 폴링 타임아웃 후 상태 고착 (2차 원인)

**파일**: `Briefly/Briefly/Views/ItemDetailView.swift`

폴링 타임아웃(70초) 후:
- `summary == nil` 그대로 + 상태 저장 없음 → 다음 오픈 시 70초 재폴링 무한 반복
- 유저 입장에서 재시도 방법 없음
- 과거 저장 항목(요약 없음)도 재시도 방법 없음

### 2-3. liveItem 갱신 체인 불안정 (3차 원인 — 초기 구현에서 확인)

초기 구현에서 `summaryStatus` 변경 후 알림(`fetchCoordinatorDidUpdate`) 발송으로 `liveItem`을 갱신하려 했으나, `items`는 뷰 생성 시점의 스냅샷이므로 현재 세션에서는 `currentItem.summaryStatus`가 업데이트되지 않아 버튼이 표시되지 않았다.

**해결 방향**: `summaryStatus`를 UI 분기 조건에서 제거하고, `isSummaryLoading` + `summary` 두 가지 로컬 상태만으로 UI를 결정한다. `summaryStatus`는 "다음 세션에서 자동 재폴링 스킵" 용도로만 사용한다.

## 4. 구현 계획

### 4-1. SavedItem — SummaryStatus 추가

**파일**: `Briefly/Briefly/Models/SavedItem.swift`

```swift
enum SummaryStatus: String, Codable {
    case unknown    // 초기 상태 (기존 데이터 하위 호환용)
    case failed     // 타임아웃 초과 — 다음 세션 자동 재폴링 스킵
    case done       // 요약 완료
}
```

커스텀 `init(from:)` 추가: `summaryStatus` 필드 없는 기존 JSON → `.unknown` 할당.

> **Xcode 등록**: `SavedItem.swift`는 Briefly + BrieflyShareExtension 양쪽에 이미 등록됨. 신규 파일 없음.

### 4-2. FetchCoordinator — 순차 → 병렬 처리

**파일**: `Briefly/Briefly/Services/FetchCoordinator.swift`

```swift
// 최대 3개 동시 처리 — B가 느려도 C, D는 독립적으로 진행
await withTaskGroup(of: Void.self) { group in
    var active = 0
    for item in pending {
        if active >= 3 {
            await group.next()
            active -= 1
        }
        let capturedItem = item
        group.addTask { await self.fetchOne(item: capturedItem) }
        active += 1
    }
}
```

**변경 전/후 비교** (4개 항목 A/B/C/D, B가 30초 소요):

| | 순차 (변경 전) | 병렬 (변경 후) |
|---|---|---|
| A rescan 호출 | t=0s | t=0s |
| B rescan 호출 | t=30s | t=0s |
| C rescan 호출 | t=60s | t=0s |
| D rescan 호출 | t=90s | t=30s |

### 4-5. ItemDetailView — UI 단순화 + 폴링 로직

**파일**: `Briefly/Briefly/Views/ItemDetailView.swift`

#### 핵심 설계 원칙

UI 분기는 **`isSummaryLoading`과 `summary` 두 가지만** 사용한다.  
`summaryStatus`는 UI에서 직접 읽지 않는다.

```
summary 존재          → 요약 텍스트 표시
isSummaryLoading      → "요약 생성 중..." 스피너
그 외 (summary 없음 + 로딩 아님)
                      → "AI 요약 다시 시도" 버튼
```

"그 외" 케이스에 해당하는 항목:
- 폴링 타임아웃 후 (`defer`로 `isSummaryLoading = false`)
- 과거 저장 항목 (summary 없음, 자동 폴링 스킵)
- `serverContentId` 없는 항목 (task guard 즉시 return)
- `summaryStatus == .failed` 항목 (task guard 즉시 return)

#### 상태 변수

```swift
@State private var summaryPollId = UUID()   // 변경 시 폴링 task 재시작
@State private var isRetrying = false        // retry 버튼 탭 시 task guard 우회용
```

#### `.task(id: summaryPollId)` 구조

```swift
.task(id: summaryPollId) {
    loadedSummary = nil
    let retrying = isRetrying
    isRetrying = false

    // summary 이미 있으면 아무것도 하지 않음 (버튼도, 로딩도 표시 안 됨)
    guard currentItem.summary == nil else { return }
    // serverContentId 없거나 토큰 없으면 즉시 return
    // → isSummaryLoading 미설정 → 버튼 표시
    guard let contentId = currentItem.serverContentId,
          let token = AuthTokenStore.shared.accessToken else { return }
    // 이전 세션에서 실패한 항목은 재시도 버튼 탭 시에만 폴링
    guard currentItem.summaryStatus != .failed || retrying else { return }

    // 여기서부터 실제 폴링 — 로딩 스피너 표시
    isSummaryLoading = true
    defer { isSummaryLoading = false }  // 성공/실패/취소 모두 로딩 해제

    for attempt in 0..<15 {
        if attempt > 0 {
            try? await Task.sleep(nanoseconds: 5_000_000_000)
        }
        if Task.isCancelled { return }
        if let detail = try? await BrieflyAPI.shared.fetchContentDetail(
            contentId: contentId, token: token),
           let summary = detail.summary {
            loadedSummary = summary
            StorageService.shared.updateSummary(for: currentItem.id, summary: summary)
            StorageService.shared.updateSummaryStatus(for: currentItem.id, status: .done)
            return
        }
    }

    // 70초 타임아웃 — summaryStatus 저장 (다음 세션에서 자동 재폴링 방지)
    // UI는 defer로 isSummaryLoading = false → 버튼이 자동으로 표시됨
    StorageService.shared.updateSummaryStatus(for: currentItem.id, status: .failed)
}
```

#### AI 요약 섹션 UI

```swift
if let summary = loadedSummary ?? currentItem.summary {
    // 요약 있으면 표시
    Text(summary)...
} else if isSummaryLoading {
    // 폴링 중
    HStack { ProgressView(); Text("요약 생성 중...") }
} else {
    // summary 없음 + 로딩 아님 → 재시도 버튼
    VStack(alignment: .leading, spacing: 8) {
        Text("요약을 가져오지 못했습니다.")
        Button(action: retryAISummary) {
            Label("AI 요약 다시 시도", systemImage: "arrow.clockwise.circle")
        }
    }
}
```

#### retryAISummary()

```swift
private func retryAISummary() {
    guard let contentId = currentItem.serverContentId,
          let token = AuthTokenStore.shared.accessToken else { return }

    StorageService.shared.updateSummaryStatus(for: currentItem.id, status: .unknown)

    Task {
        if let pageText = currentItem.articleText, pageText.count >= 200 {
            try? await BrieflyAPI.shared.rescan(
                contentId: contentId, pageText: pageText, token: token, force: true
            )
        }
    }

    isRetrying = true
    summaryPollId = UUID()  // task 재시작 → isSummaryLoading = true
}
```

## 5. UI 상태 전이

```
항목 저장
    │
    ▼
summaryStatus = .unknown

    │
    ▼ ItemDetailView 오픈
    │
    ├─ serverContentId 없음 / summaryStatus == .failed
    │       isSummaryLoading 미설정 → "AI 요약 다시 시도" 버튼 즉시 표시
    │
    └─ serverContentId 있음 + .unknown
            isSummaryLoading = true → "요약 생성 중..."
                │
                ├── 70초 내 summary 획득
                │       summaryStatus = .done
                │       loadedSummary = "<요약>"  ──► 요약 표시
                │
                └── 70초 타임아웃
                        defer: isSummaryLoading = false
                        summaryStatus = .failed (스토리지 저장)
                        ──────────────────────► "AI 요약 다시 시도" 버튼 표시

버튼 탭 (retryAISummary)
    │
    summaryStatus = .unknown (reset)
    rescan 호출 (articleText ≥ 200자일 때, force: true)
    isRetrying = true + summaryPollId 새 UUID
    │
    ▼ task 재시작
    isSummaryLoading = true → "요약 생성 중..."
        ├── 성공 → summaryStatus = .done → 요약 표시
        └── 실패 → summaryStatus = .failed → 버튼 표시
```

## 7. 엣지 케이스

| 케이스 | 처리 |
|--------|------|
| `serverContentId` 없는 항목 | task guard 즉시 return → `isSummaryLoading` 미설정 → 버튼 표시 |
| `summaryStatus == .failed` 항목 (다음 세션) | task guard 즉시 return → 버튼 표시, 자동 재폴링 없음 |
| `articleText < 200`자인 항목의 재시도 | rescan 생략, 폴링만 재시작 |
| 재시도 중 토큰 만료 | rescan 실패 무시 → 폴링 진행 → 70초 후 `.failed` |
| `summary != nil` (어떤 summaryStatus든) | task 즉시 return, UI는 summary 표시 |
| 기존 항목 (`summaryStatus` 필드 없는 JSON) | 커스텀 디코더 → `.unknown` → 처음 오픈 시 자동 폴링 1회 시도 |
| 카드 스와이프 중 재시도 탭 | `currentIndex` 변경 시 task cancel → 안전 |

## 8. 미해결 — 백엔드 병렬 요약 격리

본 스펙은 iOS 클라이언트 계층에서의 격리를 다룬다.  
백엔드의 `_background_summarize`가 내부 Semaphore 또는 공유 상태로 인해 직렬화되는 경우, iOS 수정만으로는 C, D의 요약 생성 자체를 보장할 수 없다.  
백엔드 동시성 격리는 별도 이슈(ING-006 예정)로 추적한다.
