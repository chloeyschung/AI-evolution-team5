# UX-002: Swipe Actions

## User Story

As a Briefly user, I want my swipe decisions (keep/discard) to be persisted and retrievable, so that I can build curated collections of content over time and access my saved items at any point.

## Dependencies

- **depends_on:** `@specs/UX-001.md` (Card stack provides the swipe interaction), `@specs/DAT-001.md` (Storage engine provides persistence)
- **dependency code:** `src/data/repository.py`, `src/api/routes.py`
- **depended_by:** None (enables UX-001 card stack to complete the swipe loop)

## Implementation Contract

### Level 1: Natural Language

Swipe Actions is the persistence layer for user decisions made in the swipe card stack. When a user swipes right (keep) or left (discard), this feature:

1. Records the decision with a timestamp
2. Links the decision to the specific content item
3. Provides retrieval of kept and discarded content collections
4. Supports batch recording for efficient rapid swiping
5. Exposes content statistics for UI progress indicators

The system must:
- Persist swipe actions atomically (all succeed or all fail)
- Return accurate statistics for pending/kept/discarded counts
- Support pagination for large kept/discard collections
- Maintain backward compatibility with single swipe API

### Level 2: Logic Flow (Pseudocode)

```
INPUT: Swipe gesture (content_id, action) from UX-001
PROCESS:
  1. Validate content_id exists and action is KEEP or DISCARD.
  2. Create SwipeHistory record with timestamp.
  3. Commit transaction atomically.
  4. Return created record ID for confirmation.

INPUT: Request for kept/discard content
PROCESS:
  1. JOIN Content with SwipeHistory on content_id.
  2. Filter by action (KEEP or DISCARD).
  3. Order by swiped_at DESC (most recent first).
  4. Apply pagination (limit, offset).
  5. Return Content objects with swipe metadata.

INPUT: Request for stats
PROCESS:
  1. Count all content items.
  2. Count items with KEEP action.
  3. Count items with DISCARD action.
  4. Calculate pending = all - kept - discarded.
  5. Return {pending, kept, discarded}.

OUTPUT: Persisted swipe history, filtered content lists, statistics
ERROR: ValidationError (invalid action), NotFoundError (content not found)
```

## Implementation Notes

- **Backward Compatibility:** Existing single-swipe `POST /swipe` calls must continue to work. Detect batch vs single by checking if payload contains `actions` key.
- **Transaction Atomicity:** Batch swipe must be all-or-nothing. Use single database transaction for all batch records.
- **Query Performance:** JOIN operations on SwipeHistory are indexed by content_id and action. Add composite indexes if queries become slow with large datasets.
- **Pagination:** Both limit and offset are supported for kept/discard endpoints. Default limit is 50, maximum is 100.
- **Stats Calculation:** Pending count is derived (all - kept - discarded) rather than cached, ensuring accuracy.
