# UX-002: Swipe-Based Content Processing

**Status**: Implemented | **Created**: 2026-04-14 | **Author**: abraxaspark
**F-xxx Mapping**: F-009 (Swipe View) | **Phase**: Phase 1 - MVP (Mobile Focus) | **Priority**: Critical

---

## 1. Overview

**Problem**: Users need an intuitive, low-friction way to process large volumes of saved content without the cognitive overhead of traditional folder/tag management.

**Solution**: Swipe gesture interface (Tinder-like) for binary keep/discard decisions with optional detail view for context.

**Goals**: Fast content triage, muscle memory development, decision tracking for analytics

**Non-Goals**: Multi-action swipes (MVP is binary only), swipe direction customization, undo swipe

## 3. User Story / Behavior

As a Briefly user, I want to quickly process my saved content with simple swipe gestures, so that I can triage large volumes without getting bogged down in decisions.

### Key Behaviors

- Card displays title, platform icon, thumbnail, summary preview
- Right swipe: card animates right, commits "keep", next card appears
- Left swipe: card animates left, commits "discard", content archived, next card appears
- Tap: pause swipe deck, navigate to detail view (UX-003)
- Visual indicators show intent (green tint for keep, red for discard)
- Haptic feedback confirms action commit

## 5. API Design

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/swipe` | POST | Record swipe action |
| `/api/v1/content/{id}/status` | PATCH | Update content status (for discard) |

### Request/Response Examples

**POST /api/v1/swipe**

Request:
```json
{
  "content_id": 1,
  "action": "keep"
}
```

Response (201 Created):
```json
{
  "id": 1,
  "content_id": 1,
  "action": "keep",
  "swiped_at": "2024-01-15T14:30:00Z"
}
```

**PATCH /api/v1/content/{id}/status**

Request:
```json
{
  "status": "archived"
}
```

Response (200 OK):
```json
{
  "id": 1,
  "status": "archived",
  "updated_at": "2024-01-15T14:30:00Z"
}
```

## 6. Implementation

### Files

- `src/ui/swipe_deck.dart` - Swipe deck widget with gesture detection
- `src/ui/swipe_card.dart` - Individual card with visual feedback
- `src/data/models.py` - SwipeHistory model
- `src/data/repository.py` - SwipeRepository
- `src/api/routes.py` - Swipe endpoint

### Key Logic

1. **Gesture Detection**:
   - Use GestureDetector or flutter_swipe library
   - Track dx, dy for swipe direction and distance
   - Commit threshold: 100px horizontal movement
   - Cancel if released before threshold

2. **Visual Feedback**:
   - Rotate card based on swipe distance (max 15 degrees)
   - Apply opacity overlay (green/red tint based on direction)
   - Show keep/discard icon overlay that scales with swipe distance

3. **Animation**:
   - On commit: animate card off-screen in swipe direction
   - Duration: 300ms with ease-out curve
   - Next card slides in from right with stagger (50ms)

4. **Action Recording**:
   - Create SwipeHistory record with action and timestamp
   - If action=discard: update Content status to ARCHIVED
   - Both operations should complete before next card appears

5. **Haptic Feedback**:
   - Use HapticFeedback.lightImpact() on commit
   - Trigger at same time as animation starts

### Dependencies

**Requires** (satisfied by):
- UX-001 - Card data display

**Provides** (for future):

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Swipe cancelled (< 100px) | Card springs back to center |
| Vertical swipe | Ignored (only horizontal) |
| Rapid successive swipes | Queue animations, maintain fluidity |
| Network error on record | Show error toast, allow retry, card stays |
| Empty inbox | Show "All caught up!" state |
| Tap during animation | Wait for animation to complete |
