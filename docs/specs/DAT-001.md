# DAT-001: Hybrid Storage Engine

## User Story
As a knowledge consumer, I want my saved content and decisions to be reliably stored and synchronized across devices, so that I can start on mobile and continue on desktop without losing progress.

## Implementation Contract

### Level 1: Natural Language
Hybrid Storage Engine provides persistent storage for content metadata and user interactions. It uses a cloud-first architecture where a centralized database serves as the single source of truth, with local caching for offline access. The service exposes a RESTful API for content CRUD operations and swipe action recording.

The service must:
- Store content metadata (platform, type, source URL, extracted information)
- Record user swipe actions (keep/discard decisions) with timestamps
- Provide efficient querying for content retrieval and filtering
- Handle duplicate content (same URL shared multiple times) via upsert semantics

### Level 2: Logic Flow (Pseudocode)
```
INPUT: API Request (HTTP)
  - CREATE: content metadata object
  - READ: query parameters (limit, offset, filters)
  - UPDATE: content identifier + new metadata
  - SWIPE: content identifier + action type

PROCESS (Create/Update Content):
  1. Validate incoming metadata against schema.
  2. Check for existing content by source URL.
  3. If exists: update metadata fields (upsert).
  4. If not exists: insert new record with timestamp.
  5. Persist to database.
  6. Return created/updated content object.

PROCESS (List Content):
  1. Apply query filters and pagination.
  2. Order by recency (newest first).
  3. Fetch from database.
  4. Return content list.

PROCESS (Record Swipe):
  1. Validate content exists and action is valid.
  2. Create swipe history record with timestamp.
  3. Persist to database.
  4. Return confirmation.

OUTPUT: API Response (JSON)
  - Content object with identifier and metadata
  - Swipe confirmation with timestamp
ERROR: ValidationError (schema violation), NotFoundError (missing content), DatabaseError (persistence failure)
```

## Verification Contract

### Level 1: Scenarios (Natural Language)
- **Success:** New content metadata is saved and assigned a unique identifier.
- **Success:** Content with existing URL is updated (upsert) rather than duplicated.
- **Success:** Content list is retrieved ordered by creation date (newest first).
- **Success:** Swipe action is recorded with timestamp for a valid content.
- **Failure:** Invalid metadata (missing required fields) is rejected with validation error.
- **Edge:** Pagination parameters are respected when listing content.
- **Edge:** Multiple swipe actions for same content are all recorded (history preserved).

### Level 2: Given-When-Then
```
GIVEN valid content metadata with a new URL
WHEN save_content is called
THEN a new Content record is created with a unique ID and created_at timestamp

GIVEN content metadata with an existing URL
WHEN save_content is called
THEN the existing record is updated and returned (no duplicate created)

GIVEN multiple content records in the database
WHEN get_contents is called with limit=10
THEN at most 10 records are returned, ordered by created_at descending

GIVEN a valid content ID and swipe action
WHEN record_swipe is called
THEN a SwipeHistory record is created with the action and current timestamp

GIVEN content metadata missing required fields (e.g., URL)
WHEN save_content is called
THEN a ValidationError is raised

GIVEN multiple swipe actions for the same content
WHEN record_swipe is called multiple times
THEN each action creates a separate SwipeHistory record (history preserved)
```

## Implementation Notes
- Use upsert semantics (insert or update by URL) to handle duplicate shares gracefully.
- All timestamps should be timezone-aware (UTC) for consistent cross-device synchronization.
- The repository pattern abstracts database operations; the API layer handles HTTP routing and validation.
- Swipe history is separate from content to allow multiple actions per content and efficient querying.
- Cloud database is PostgreSQL; local caching strategy is out of scope for MVP (Phase 2).
