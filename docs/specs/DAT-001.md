# DAT-001: Hybrid Storage Engine

**Status**: Implemented | **Created**: 2026-04-14 | **Author**: abraxaspark
**F-xxx Mapping**: F-018 (iOS ↔ Web Sync), F-012 (Content Status) | **Phase**: Phase 1 - MVP (Mobile Focus) | **Priority**: Critical

---

## 1. Overview

**Problem**: Users need reliable storage and synchronization of saved content across devices without losing progress.

**Solution**: Cloud-first storage engine with content metadata persistence, swipe action recording, and upsert semantics for duplicate handling.

**Goals**: Reliable content storage, efficient querying, duplicate prevention, status tracking (INBOX/ARCHIVED)

**Non-Goals**: Local caching strategy (Phase 2), offline-first architecture, conflict resolution

---

## 3. User Story / Behavior

As a knowledge consumer, I want my saved content and decisions to be reliably stored, so that I can start on mobile and continue on desktop without losing progress.

### Key Behaviors

- New content saved with metadata, defaults to INBOX status
- Duplicate URLs update existing record (upsert)
- Swipe actions recorded with timestamps
- "Discard" swipe automatically sets status to ARCHIVED
- Content list ordered by recency (newest first)

## 5. API Design

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/content` | POST | Create/update content (upsert by URL) |
| `/api/v1/content` | GET | List content with optional status filter |
| `/api/v1/content/{id}` | GET | Get content by ID |
| `/api/v1/content/{id}/status` | PATCH | Update content status |
| `/api/v1/swipe` | POST | Record swipe action |

### Request/Response Examples

**POST /api/v1/content**

Request:
```json
{
  "platform": "Medium",
  "content_type": "article",
  "url": "https://medium.com/article/123",
  "title": "Article Title",
  "summary": "AI-generated summary..."
}
```

Response (201 Created):
```json
{
  "id": 1,
  "platform": "Medium",
  "content_type": "article",
  "url": "https://medium.com/article/123",
  "title": "Article Title",
  "status": "inbox",
  "created_at": "2024-01-01T12:00:00Z"
}
```

**POST /api/v1/swipe**

Request:
```json
{
  "content_id": 1,
  "action": "keep"
}
```

Response (201 Created):
```json
{
  "id": 1,
  "content_id": 1,
  "action": "keep",
  "swiped_at": "2024-01-01T12:05:00Z"
}
```

## 6. Implementation

### Files

- `src/data/models.py` - Content, SwipeHistory models with ContentStatus enum
- `src/data/repository.py` - ContentRepository, SwipeRepository
- `src/api/routes.py` - Content and swipe endpoints
- `src/api/schemas.py` - ContentCreate, ContentResponse, SwipeCreate schemas

### Key Logic

1. **Upsert by URL**:
   - Check if content exists by URL
   - If exists: update metadata fields, preserve status
   - If not: insert new record with status=INBOX

2. **Swipe Recording**:
   - Validate content exists and action is valid
   - Create SwipeHistory record with timestamp
   - If action=discard: update content status to ARCHIVED

3. **Status Transition**:
   - Only INBOX → ARCHIVED allowed (one-way)
   - Raise error if attempting ARCHIVED → INBOX

4. **Content Listing**:
   - Order by created_at DESC (newest first)
   - Optional status filter (INBOX or ARCHIVED)
   - Pagination with limit/offset

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Duplicate URL | Update existing record (upsert) |
| Missing required fields | ValidationError |
| Content not found | NotFoundError |
| Invalid status transition | ValidationError (one-way only) |
| Multiple swipes on same content | All recorded (history preserved) |
