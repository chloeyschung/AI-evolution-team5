## 1. Overview

### 1.1. Problem Statement

Users have accumulated content in their Archive (kept items) but lack a way to rediscover relevant content based on their interests and consumption patterns. A static list doesn't help users find content that matches their current interests or trending topics within their saved collection.

### 1.2. Solution

A personalized "Trend Feed" that curates and ranks archived content based on:
- User's swipe history (kept vs. discarded patterns)
- Interest tags (user-defined preferences)
- Content tags (AI-generated categories)
- Recency and engagement signals

### 1.3. Goals

- Help users rediscover relevant content from their archive
- Surface trending topics based on user's consumption patterns
- Provide a dynamic, personalized viewing experience beyond static lists
- Increase content re-engagement rate

### 1.4. Non-Goals

- Community/social trending (other users' content) - Phase 4+
- Real-time external trend integration - Phase 4+
- Content recommendation from external sources - Phase 3+

## 3. Algorithm Design

### 3.1. Relevance Score Formula

```
relevance_score = (
    interest_match_weight * interest_match_score +
    tag_similarity_weight * tag_similarity_score +
    recency_weight * recency_score +
    engagement_weight * engagement_score
)
```

**Default Weights**:
- `interest_match_weight`: 0.35 (35%)
- `tag_similarity_weight`: 0.30 (30%)
- `recency_weight`: 0.20 (20%)
- `engagement_weight`: 0.15 (15%)

### 3.2. Component Scores

#### 3.2.1. Interest Match Score (0-1)

```
interest_match_score = matched_interests / total_user_interests
```

- `matched_interests`: Count of user interest tags that match content tags
- `total_user_interests`: Total number of user's interest tags
- If user has no interests, score = 0.5 (neutral)

#### 3.2.2. Tag Similarity Score (0-1)

```
tag_similarity_score = jaccard_similarity(content_tags, preferred_tags)
```

- `jaccard_similarity(A, B) = |A ∩ B| / |A ∪ B|`
- `preferred_tags`: Tags from user's most-kept content (top 20 by keep count)
- If no preferred tags, score = 0.5 (neutral)

#### 3.2.3. Recency Score (0-1)

```
recency_score = 1 / (1 + days_since_kept / half_life_days)
```

- `days_since_kept`: Days between now and when content was kept
- `half_life_days`: 30 (content loses half its recency score after 30 days)
- This creates a smooth decay curve

#### 3.2.4. Engagement Score (0-1)

```
engagement_score = keep_ratio_in_same_category
```

- `keep_ratio_in_same_category`: Ratio of kept vs. total content with same tags
- Calculated per tag, then averaged across all content tags
- If user kept 8/10 "technology" tagged items, tech content gets 0.8 engagement score

### 3.3. Minimum Threshold

Content must have `relevance_score >= 0.1` to appear in trend feed. This prevents very low-relevance content from appearing.

## 5. API Design

### 5.1. Endpoint

```
GET /content/trend-feed
```

### 5.2. Query Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `limit` | int | 20 (1-50) | Maximum items to return |
| `offset` | int | 0 (>=0) | Pagination offset |
| `time_range` | string | "all" | Filter: "week", "month", "all" |
| `min_score` | float | 0.1 (0-1) | Minimum relevance score threshold |

### 5.3. Response Schema

```python
class TrendFeedItem(BaseModel):
    """Item in trend feed."""
    model_config = {"from_attributes": True}

    content: ContentResponse  # Existing schema
    relevance_score: float  # 0-1
    matched_interests: List[str]  # User interests that matched
    top_tags: List[str]  # Top contributing tags (max 3)


class TrendFeedResponse(BaseModel):
    """Trend feed response."""
    items: List[TrendFeedItem]
    total: int  # Total matching items (for pagination)
    has_more: bool  # Whether more items available
```

### 5.4. Example

```
GET /content/trend-feed?limit=10&time_range=month

Response: { items: [...TrendFeedItem], total: 45, has_more: true }
// Each item: { content, relevance_score, matched_interests, top_tags }
```

### 6.3. Performance Considerations

1. **Caching**: Cache user's preferred tags (recalculate only when swipe history changes significantly)
2. **Batch Queries**: Fetch all needed data in minimal queries (avoid N+1)
3. **Indexing**: Ensure `ContentTag.tag` and `InterestTag.tag` are indexed
4. **Lazy Loading**: Only calculate scores for items that might pass the threshold

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| User has no archived content | Return empty list with message "Keep some content first" |
| User has < 10 archived items | Return all kept items sorted by recency (no scoring) |
| User has no interest tags | Use tag_similarity and engagement only (interest_match = 0.5) |
| Content has no tags | Use recency and engagement only (tag_similarity = 0.5) |
| All scores below threshold | Return empty list (no low-quality recommendations) |
