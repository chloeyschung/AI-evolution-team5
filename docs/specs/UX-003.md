# UX-003: Summary Detail View

**Status**: Implemented | **Created**: 2026-04-14 | **Author**: abraxaspark
**F-xxx Mapping**: F-012 (Content Detail View) | **Phase**: Phase 1 - MVP (Mobile Focus) | **Priority**: High

---

## 1. Overview

**Problem**: Users need to access full content details and metadata when the card preview is insufficient for decision-making or reference.

**Solution**: On-demand detail view displaying complete content information, metadata, swipe history, and status with direct source navigation.

**Goals**: Context expansion, source access, status management (INBOX/ARCHIVED), graceful metadata handling

**Non-Goals**: Content editing, tag management, social sharing (MVP scope)

## 3. User Story / Behavior

As a Briefly user, I want to view expanded details about content I've saved, so that I can access the original source or review metadata when I need more context beyond the card preview.

### Key Behaviors

- Access from card tap (UX-001) or collection item tap (kept/discarded)
- Full URL displayed as clickable link to source
- Complete metadata shown in structured format
- Status badge indicates INBOX or ARCHIVED
- "Mark as read" button: active for INBOX, dimmed for ARCHIVED
- Swipe history shown if content was previously swiped
- Missing metadata fields handled gracefully (hidden or "N/A")

## 5. API Design

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/content/{id}` | GET | Get full content detail |
| `/api/v1/content/{id}/status` | PATCH | Update content status (mark as read) |

### Request/Response Examples

**GET /api/v1/content/{id}**

Response (200 OK):
```json
{
  "content_id": 1,
  "url": "https://example.com/article/123",
  "summary": "3-line AI-generated summary here...",
  "metadata": {
    "platform": "Medium",
    "content_type": "article",
    "title": "Article Title",
    "author": "John Doe",
    "published_at": "2024-01-15T10:30:00Z"
  },
  "status": "inbox",
  "ingested_at": "2024-01-15T12:00:00Z",
  "swipe_history": {
    "action": "keep",
    "swiped_at": "2024-01-15T14:00:00Z"
  }
}
```

Response (404 Not Found):
```json
{
  "detail": "Content not found"
}
```

**PATCH /api/v1/content/{id}/status**

Request:
```json
{
  "status": "archived"
}
```

Response (200 OK):
```json
{
  "content_id": 1,
  "status": "archived",
  "updated_at": "2024-01-15T15:00:00Z"
}
```

Response (400 Bad Request):
```json
{
  "detail": "Content is already archived"
}
```

## 6. Implementation

### Files

- `src/api/schemas.py` - ContentDetailResponse schema
- `src/api/routes.py` - GET /content/{id}, PATCH /content/{id}/status endpoints
- `src/data/repository.py` - ContentRepository.get_by_id(), update_status()

### Key Logic

1. **Detail Fetch**:
   - Fetch Content object by ID
   - JOIN with SwipeHistory to get action and timestamp (if exists)
   - Return null for missing optional fields (author, title, published_at)
   - Include status field in response

2. **Status Transition**:
   - Validate current status is INBOX
   - Update to ARCHIVED (one-way transition)
   - Raise ValidationError if already ARCHIVED
   - Return updated content with new timestamp

3. **URL Security**:
   - Validate URL scheme (http/https only) before external navigation
   - Prevent malicious redirects

4. **Error Handling**:
   - Content not found → 404 NotFoundError
   - Invalid status transition → 400 ValidationError
   - Network error → retry option on frontend

### Dependencies

**Requires** (satisfied by):
- UX-001 - Card stack navigation (entry point)
- AI-002 - Metadata extraction (platform, type, author, timestamp)
- DAT-001 - Content storage with status field

**Provides** (for future):
- UX-004 - Status filter for inbox/archive lists
- INT-001 - Share from detail view

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Content not found | 404 error with user-friendly message |
| Missing metadata fields | Return null, frontend hides or shows "N/A" |
| No swipe history | swipe_history field is null |
| Invalid status transition | 400 error (ARCHIVED → INBOX not allowed) |
| Network error | Frontend shows retry option |
