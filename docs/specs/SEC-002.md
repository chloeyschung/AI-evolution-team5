# SEC-002: YouTube OAuth State CSRF Protection

**Status**: Spec | **Created**: 2026-04-16 | **Author**: abraxaspark
**F-xxx Mapping**: N/A (security hardening) | **Phase**: Stabilization | **Priority**: High

---

## 1. Overview

**Problem**: The YouTube OAuth connect/callback flow uses `state=str(user_id)` — a predictable integer — which allows any caller who knows a victim's user ID to forge a callback request and link their own YouTube account to the victim's profile (CSRF).

**Solution**: Replace the state parameter with a cryptographically random token generated at connect time, stored server-side with a short TTL, and consumed (deleted) on first use at callback time.

**Goals**: OAuth CSRF attack surface eliminated; legitimate YouTube connect flow unaffected.

**Non-Goals**: Does not cover LinkedIn OAuth (LinkedIn uses manual import only — no connect/callback routes exist). Does not introduce PKCE. Does not rotate tokens after each sync.

---

## 2. Requirements

| ID | Requirement | Priority |
|----|-------------|----------|
| FR-1 | `connect_youtube` generates a `secrets.token_urlsafe(16)` state token instead of `str(user_id)` | P0 |
| FR-2 | State token stored in new `OAuthState` table with `user_id`, `provider`, `expires_at` (15-min TTL) | P0 |
| FR-3 | `youtube_callback` looks up `state` in `OAuthState` table scoped to `provider=youtube` | P0 |
| FR-4 | `youtube_callback` derives `user_id` from the `OAuthState` record, not from the `state` query param | P0 |
| FR-5 | `OAuthState` record deleted immediately after first valid use (single-use semantics) | P0 |
| FR-6 | Missing, expired, or already-consumed state tokens → `HTTP 400` + `ErrorCode.INVALID_STATE` | P0 |
| NFR-1 | State token TTL is 15 minutes from creation | P0 |
| NFR-2 | `OAuthState.state_token` has a DB-level unique index | P1 |

---

## 3. User Story / Behavior

As a Briefly user, I want to connect my YouTube account securely, so that an attacker cannot hijack my OAuth callback to overwrite my integration with their own account.

### Key Behaviors

- Clicking "Connect YouTube" calls `POST /integrations/youtube/connect`, which generates a random token, stores it, and returns the OAuth URL with `state=<random_token>`.
- Completing the Google consent screen sends the user to `GET /integrations/youtube/callback?code=...&state=<random_token>`.
- The callback verifies the token exists in `OAuthState`, is not expired, is scoped to `youtube`, then atomically consumes it and proceeds with token exchange.
- If the `state` value is unknown, expired, or already consumed, the callback returns `400 invalid_state` with no token exchange attempted.

## 4. Data Models

### New Tables

**`oauth_states`**: `id` (int PK), `user_id` (int, not null, indexed), `provider` (varchar 50, indexed), `state_token` (varchar 64, unique, indexed), `expires_at` (datetime, indexed), `created_at` (datetime, default utc_now)

### Existing Used

- `IntegrationTokens` — stores YouTube access/refresh tokens after successful OAuth
- `IntegrationRepository` — extended with `save_oauth_state` + `get_and_consume_oauth_state`

---

## 5. API Design

No new endpoints. Two existing endpoints are modified:

| Endpoint | Method | Change |
|----------|--------|--------|
| `/integrations/youtube/connect` | POST | `state` changes from `str(user_id)` to `token_urlsafe(16)`; new `OAuthState` row written |
| `/integrations/youtube/callback` | GET | `state` validated against `OAuthState` table; `user_id` read from record |

### Request/Response

**`POST /integrations/youtube/connect` Response (200 OK) — unchanged externally:**
```json
{ "auth_url": "https://accounts.google.com/o/oauth2/v2/auth?...&state=<opaque_token>" }
```

**`GET /integrations/youtube/callback` Response (200 OK) — unchanged externally:**
```json
{ "message": "Connected to YouTube successfully", "user_id": 42 }
```

**`GET /integrations/youtube/callback` Error (400):**
```json
{ "detail": "invalid_state" }
```

## 6. Implementation

### Files

- `src/data/models.py` — Add `OAuthState` model; add to `__all__` exports
- `src/integrations/repositories/integration.py` — Add `save_oauth_state()` and `get_and_consume_oauth_state()` to `IntegrationRepository`; import `OAuthState`
- `src/api/routes.py` — Update `connect_youtube` (~line 1708) and `youtube_callback` (~lines 1740–1744)
- `tests/integrations/test_youtube.py` — Add tests for state generation, validation, expiry, and single-use enforcement

### Key Logic

1. **connect_youtube**: `state_token = secrets.token_urlsafe(16)` → `repo.save_oauth_state(user_id, "youtube", state_token, utc_now() + timedelta(minutes=15))` → embed in `auth_url`.
2. **youtube_callback**: `user_id = await repo.get_and_consume_oauth_state(state, "youtube")` → if `None`, raise `400 INVALID_STATE` → proceed with token exchange using returned `user_id`.
3. **get_and_consume_oauth_state**: `SELECT WHERE state_token=? AND provider=? AND expires_at > now()` → if found, `DELETE WHERE id=?`, return `user_id` → else return `None`. Both ops in the same DB session (auto-committed by route handler).

### Dependencies

**Requires** (satisfied):
- `INT-001` — `IntegrationTokens` model and `IntegrationRepository` both exist
- `ErrorCode.INVALID_STATE` — already defined in `src/constants.py:146`

**Provides** (for future):
- Pattern reusable if LinkedIn ever adds an OAuth connect flow

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| `state` not found in `OAuthState` table | `400 invalid_state` |
| `state` found but `expires_at` < now | `400 invalid_state` |
| `state` already consumed (replay) | `400 invalid_state` (record deleted on first use) |
| `state` valid but wrong provider | `400 invalid_state` (provider scoped in query) |
| YouTube code exchange fails after valid state | `400 youtube_auth_failed`; state consumed, user must restart flow |
| User calls connect twice rapidly | Two `OAuthState` rows; only the one matching the completed callback is consumed |

---

## 8. Testing

- **Unit**: `get_and_consume_oauth_state` returns `None` for unknown token, expired token, wrong provider; returns `user_id` and deletes row for valid token.
- **Integration**: Full connect → callback flow stores tokens; forged callback (`state=42`) returns `400`; replay of consumed state returns `400`.
- **Acceptance**: After fix, `state` in OAuth URL is a non-integer opaque string; `youtube_callback` rejects the old integer pattern.

---

## 9. Sensory Verification

- **Visual (시각)**: After `connect_youtube`, an `OAuthState` row exists in DB with `state_token` matching the value in `auth_url`. After successful `youtube_callback`, the row is gone and an `IntegrationTokens` row exists.
- **Auditory (청각)**: `GET /integrations/youtube/callback?code=fake&state=42` returns `{"detail": "invalid_state"}` with HTTP 400.
- **Tactile (촉각)**: End-to-end YouTube OAuth flow completes without error; tokens stored; subsequent sync can be triggered.
