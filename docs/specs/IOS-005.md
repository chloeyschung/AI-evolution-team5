# IOS-005: 로그인 후 로컬 저장 항목 서버 업로드 (Post-Login Sync)

**Status**: In Progress (서버 설정 대기 중) | **Created**: 2026-05-09 | **Author**: suyoung
**F-xxx Mapping**: Post-login local→server sync | **Phase**: Phase 3 - Sync | **Priority**: Medium

---

## 1. Overview

**Problem**: 로그인 전에 Share Extension으로 저장한 링크들은 로컬 UserDefaults에만 존재하고 `serverContentId`가 없다. 이후 로그인을 해도 이 항목들은 서버(웹 대시보드)에 보이지 않는다.

**Solution**: 로그인 성공 직후, 로컬에 저장된 항목 중 `serverContentId == nil`인 것들을 `POST /api/v1/share`로 일괄 업로드하고 반환받은 서버 ID를 로컬에 저장한다.

**Goals**:
- 로그인 전 저장된 링크가 로그인 후 웹 대시보드에 표시된다
- Keep/Delete 처리 시 swipe API가 정상 호출된다 (`serverContentId` 보유)
- 이미 서버에 있는 항목은 중복 저장되지 않는다 (서버가 URL 중복 처리)

**Non-Goals**:
- 서버 → 로컬 역방향 동기화 (IOS-006+ 별도 티켓)
- 실시간 동기화 (로그인 시 1회 + 포그라운드 진입 시 실행)

## 3. User Story / Behavior

**As a** 로그인 전에 이미 링크를 저장한 Briefly 사용자,  
**I want to** 로그인하고 나면 기존에 저장한 링크가 웹 대시보드에서도 보이길 원한다,  
**so that** 어떤 기기에서든 내 저장 목록을 확인하고 Keep/Delete 처리를 이어서 할 수 있다.

### Key Behaviors

```
[로그인 성공] → AccountView.loginSuccess 콜백
             → SyncService.syncLocalItemsToServer() 호출
             → 로컬 항목 중 serverContentId == nil 필터링
             → 항목별 POST /api/v1/share 호출
             → 성공 시 StorageService.updateItemById(serverContentId 포함)
             → 완료 후 NotificationCenter post → Library 화면 갱신

[앱 포그라운드 진입] → ContentView.scenePhase == .active
                    → SyncService.syncLocalItemsToServer() 호출 (동일 흐름)

[Keep/Delete] → serverContentId 있으면 즉시 swipe API 호출
             → serverContentId 없으면 share → swipe 순서로 호출
```

## 4. Implementation

### Files 변경 목록

| 파일 | 변경 내용 |
|------|-----------|
| `Briefly/Services/SyncService.swift` | actor로 재작성, 토큰 만료 시 1회 갱신 후 재시도 |
| `Briefly/Services/BrieflyAPI.swift` | `refreshCurrentToken()` 헬퍼 추가 |
| `Briefly/Services/FetchCoordinator.swift` | `authSessionDidExpire` 알림 이름 정의 추가 |
| `Briefly/Views/AccountView.swift` | 로그인 성공 콜백에서 `SyncService.syncLocalItemsToServer()` 호출 |
| `Briefly/Views/LibraryView.swift` | scenePhase/onAppear에서 viewModel.reload()만 담당 (sync는 ContentView로 이관) |
| `Briefly/Views/ContentView.swift` | scenePhase/onAppear에서 SyncService 호출 |
| `Briefly/Views/ItemDetailView.swift` | Keep/Delete 시 serverContentId 없으면 share → swipe fallback |
| `BrieflyShareExtension/ShareViewController.swift` | 로그인 상태에서 공유 시 API 호출 + 에러 로깅 추가 |

### SyncService 핵심 설계

- **`actor SyncService`**: 동시 호출을 직렬화해 토큰 갱신 race condition 방지
- **`isRunning` 플래그**: actor가 `await` 지점에서 suspend될 때 두 번째 진입을 즉시 차단
- **순차 처리**: `withTaskGroup` 대신 `for` 루프로 변경 — `didRefreshToken` 상태를 아이템 간에 공유
- **토큰 갱신**: 401 수신 시 `refreshCurrentToken()` 1회 시도, 성공 시 새 토큰으로 재시도

## 5. 디버깅 기록 (2026-05-09)

### 발견된 버그들

#### Bug 1: SyncService가 로그인 시에만 호출됨
- **증상**: 로그인 후 바로 저장된 항목은 웹에서 보이지만, 이후 새 링크 저장은 웹에 반영 안 됨
- **원인**: `SyncService.syncLocalItemsToServer()`가 `AccountView` 로그인 성공 콜백에서만 호출됨
- **수정**: `ContentView.scenePhase == .active` 및 `onAppear`에서도 호출

#### Bug 2: TabView 비활성 탭에서 scenePhase onChange 미실행
- **증상**: `[SyncService]` 로그가 전혀 출력되지 않음
- **원인**: `SyncService` 호출이 `LibraryView`의 `.onChange(of: scenePhase)`에 있었는데, Library 탭이 비활성 탭이면 SwiftUI가 이 핸들러를 실행하지 않음
- **수정**: `ContentView`(TabView 부모)로 이관 — 항상 실행 보장

#### Bug 3: performKeep/performDelete에서 serverContentId 없으면 swipe 스킵
- **증상**: Share Extension에서 API 호출 실패 시 serverContentId가 없는 채로 Keep/Delete 하면 웹에 반영 안 됨
- **원인**: `swipe()` 전에 serverContentId 존재 여부만 체크하고 없으면 조용히 스킵
- **수정**: serverContentId 없으면 먼저 `share()` 호출 후 swipe

#### Bug 4: onAppear + scenePhase onChange 동시 호출로 refresh token race condition
- **증상**: `[BrieflyAPI] 토큰 갱신 실패: 서버 오류 (401)` — refresh token도 401 반환
- **원인**: 앱 최초 실행 시 `onAppear`와 `scenePhase == .active` 두 곳에서 SyncService가 거의 동시에 호출됨. 두 Task가 동일한 refresh token으로 갱신을 시도하면서 서버의 token rotation 정책에 의해 첫 번째가 소비하면 두 번째는 401
- **수정**: `SyncService`를 `actor`로 변환 + `isRunning` 플래그로 동시 진입 차단

#### Bug 5: Share Extension 프로세스 종료로 API Task kill
- **증상**: 로그인 상태에서 공유해도 serverContentId가 저장되지 않는 경우 발생
- **원인**: Share Extension의 3초 자동 닫힘 또는 사용자 닫기 시 iOS가 Extension 프로세스 종료. 진행 중인 `Task`가 kill됨
- **완화**: ContentView의 포그라운드 sync가 serverContentId 없는 항목을 재업로드

## 6. 서버 측 필수 대응 사항 (백엔드 팀 확인 필요)

> **현재 iOS sync가 정상 작동하려면 아래 서버 설정이 필요합니다.**

### 6-1. 토큰 수명 설정 (최우선)

| 토큰 | 권장 수명 | 현재 추정 |
|------|-----------|-----------|
| `access_token` | 1시간 ~ 24시간 | 너무 짧은 것으로 추정 (수 분?) |
| `refresh_token` | 30일 이상 | 즉시 만료되는 것으로 추정 |

**근거**: 로그인 직후 재실행 시 즉시 `access_token` 401 → `refresh_token` 갱신 시도 → `refresh_token`도 401. 로그인 직후 수 분 내에 두 토큰 모두 무효화되는 것으로 보임.

### 6-2. Refresh Token Rotation 정책

현재 iOS 클라이언트는 refresh token으로 갱신 시 서버가 반환하는 새 `refresh_token`을 저장합니다. 서버가 **token rotation** (1회 사용 후 무효화) 정책을 사용하는 경우:

- ✅ iOS 코드는 이를 지원함 (`AuthTokenStore`에 새 refresh_token 저장)
- ⚠️ 주의: rotation 사용 시 **동시 갱신 요청이 절대 없어야** 함 (현재 actor로 직렬화됨)
- 확인 필요: `/api/v1/auth/refresh` 엔드포인트가 동일 refresh_token으로 중복 요청 시 어떻게 처리하는지

### 6-3. POST /api/v1/share 응답 형식

iOS 클라이언트가 기대하는 응답:

```json
{
  "id": 123,
  "url": "https://example.com",
  "title": "페이지 제목",
  "platform": "web",
  "created_at": "2026-05-09T12:00:00Z"
}
```

`id` 필드가 없거나 키 이름이 다르면 디코딩 실패 → serverContentId 저장 안 됨 → 웹에 미표시.

### 6-4. POST /api/v1/swipe 응답 형식

iOS 클라이언트가 기대하는 응답:

```json
{
  "id": 456,
  "content_id": 123,
  "action": "keep"
}
```

### 6-5. 웹 대시보드 표시 조건 확인

- `/api/v1/share`만 호출(swipe 없음)한 항목이 웹에 표시되는지 확인 필요
- 현재 iOS 테스트 결과: swipe 없이 share만 해도 로그인 시 sync된 항목은 웹에 표시됨 (로그아웃→재로그인 시 확인됨)
- 그러나 로그인 유지 상태에서의 포그라운드 sync는 토큰 만료로 인해 미검증

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| 로컬 항목이 없음 | guard 조건으로 early return |
| 일부 항목 업로드 실패 | 에러 로그 후 나머지 항목 계속 처리 |
| 이미 서버에 동일 URL 존재 | 서버가 새 row로 저장 (중복 허용), serverContentId 정상 반환 |
| 로그인 전 .kept/.deleted 항목 | 업로드 후 즉시 swipe API 호출로 서버 상태 소급 적용 |
| access_token 만료 (401) | refresh_token으로 1회 갱신 시도 후 재시도 |
| refresh_token 만료 (401) | 에러 로그 후 해당 항목 skip (강제 로그아웃 없음) — 재로그인 시 자동 재시도 |
| Share Extension 프로세스 종료 | 항목은 로컬 inbox에 저장됨. 메인 앱 포그라운드 진입 시 SyncService가 재업로드 |
| 업로드 중 네트워크 없음 | 모든 항목 실패, 다음 포그라운드 진입 시 재시도 |
| SyncService 동시 호출 | actor + isRunning 플래그로 두 번째 호출 즉시 차단 |

## 8. Testing

### 수동 검증 시나리오

1. **로그인 전 저장 → 로그인 sync**
   - 로그아웃 상태에서 링크 3개 저장
   - 로그인 → `[SyncService] share 성공` 로그 확인
   - 웹 대시보드 새로고침 → 3개 링크 표시 확인

2. **로그인 상태에서 Keep/Delete → 웹 반영** *(서버 토큰 설정 수정 후 재검증 필요)*
   - 로그인 상태에서 새 링크 저장
   - 앱으로 돌아와 `[SyncService] share 성공: contentId=XXX` 로그 확인
   - Keep 또는 Delete 수행 → `[Keep] swipe 성공` / `[Delete] swipe 성공` 로그 확인
   - 웹 대시보드 Archived/Deleted 탭 확인

3. **serverContentId 없는 항목 Keep/Delete**
   - 로그아웃 상태에서 링크 저장 후 Keep
   - 로그인 → `[SyncService] share 성공` + `[SyncService] swipe keep 성공` 로그 확인

### 현재 검증 결과

| 시나리오 | 결과 | 비고 |
|----------|------|------|
| 로그인 전 저장 → 로그인 sync | ✅ 동작 | 웹 표시 확인됨 |
| 로그인 상태 저장 → 포그라운드 sync | ❌ 미검증 | refresh_token 401로 인해 서버 미연결 |
| Keep/Delete → 웹 반영 | ❌ 미검증 | 위와 동일 원인 |

## 9. Acceptance Criteria

- [ ] 로그인 직후 로컬 항목이 웹에서 보임
- [ ] 로그인 상태 유지 중 새 링크 저장 후 웹에서 보임 *(서버 토큰 수명 수정 후)*

- [ ] Delete → 웹 Deleted 탭에 표시 *(서버 토큰 수명 수정 후)*
- [ ] 30분 이상 앱 사용 후에도 재로그인 불필요 *(서버 access_token 수명 수정 후)*
