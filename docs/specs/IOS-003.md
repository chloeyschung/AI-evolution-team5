# IOS-003: Keep / Delete — 링크 보존 또는 삭제 결정 기능

**Status**: Done | **Created**: 2026-05-09 | **Author**: suyoung
**F-xxx Mapping**: Keep/Delete Flow (iOS) | **Phase**: Phase 2 - Core UX | **Priority**: High

---

## 1. Overview

**Problem**: 앱에 저장된 링크가 Inbox에 쌓이기만 하고, 사용자가 장기 보존(Keep)할 것인지 삭제(Delete)할 것인지 결정할 수 없다.

**Solution**: Inbox 상세 페이지를 카드 리더 모드로 전환하여 — 하단 텍스트 버튼 3개(Delete / Skip Now / Keep) 또는 좌우 스와이프로 한 카드씩 처리하고, 처리 후 다음 링크를 자동으로 보여준다. Library 탭은 Inbox / Archived / Deleted 3개로 분리하며, Share Extension 저장 시 서버 content_id를 캡처해 웹 대시보드와 상태를 동기화한다.

**Goals**:
- Inbox 아이템에 대해 Keep / Skip Now / Delete를 버튼 탭 또는 스와이프로 처리할 수 있다
- Skip Now는 아이템을 Inbox에 남긴 채 다음 카드로 넘어간다
- 한 카드 처리 완료 시 다음 Inbox 아이템이 카드처럼 전환되며 표시된다
- Keep/Delete 결과가 `POST /api/v1/swipe`를 통해 웹 대시보드에 반영된다
- Share Extension 저장 시 서버 content_id를 `SavedItem.serverContentId`에 저장한다

**Non-Goals**:
- 삭제(Delete) 후 즉시 영구 제거 — Deleted 탭에서 계속 조회 가능 (완전 삭제는 별도 티켓)
- Keep/Delete 결정에 대한 Undo 기능 (향후)
- Archived / Deleted 탭의 카드 리더 모드 (읽기 전용 상세 뷰만 제공)

## 3. User Story / Behavior

**As a** Briefly 사용자,  
**I want to** Inbox에 쌓인 링크를 Keep 또는 Delete로 분류하고 싶다,  
**so that** 나중에 다시 꺼내 볼 링크와 버릴 링크를 명확히 관리할 수 있다.

### Key Behaviors

- 링크 저장 직후: 아이템은 항상 Inbox에 위치 (`status = .unread`)
- Inbox 카드 탭 → 카드 리더 모드 진입 (탭한 카드부터 나머지 Inbox 순서대로)
- 카드 리더 모드 하단: **[Delete] [Skip Now] [Keep]** 텍스트 버튼 3개 고정 표시 (구분선으로 분리, 배경 없음)
- 사용자는 스크롤하여 콘텐츠를 읽고, 읽은 뒤 버튼 탭 또는 스와이프로 결정
- **Keep 처리** (버튼 탭 or 오른쪽 스와이프): 카드가 오른쪽으로 날아가고 다음 카드 표시
- **Delete 처리** (버튼 탭 or 왼쪽 스와이프): 카드가 왼쪽으로 날아가고 다음 카드 표시
- 모든 Inbox 카드를 처리하면 자동으로 Library 화면으로 복귀
- Archived / Deleted 탭의 상세 페이지: 버튼 없음, 읽기 전용

### 버튼 레이아웃 & 스와이프 방향 매핑

```
┌──────────────────────────────────────┐  ← 카드 (rounded corner + shadow)
│ [DELETE]              [KEEP]         │  ← 스와이프 방향 인디케이터 배지
│                                      │
│          콘텐츠 (스크롤)               │
│                                      │
└──────────────────────────────────────┘

┌─────────────┬─────────────┬──────────────┐
│   Delete    │  Skip Now   │    Keep      │  ← 하단 텍스트 버튼 (구분선 분리, 배경 없음)
│  (빨간색)   │  (회색)     │  (파란색)    │
└─────────────┴─────────────┴──────────────┘
← 왼쪽 스와이프 = Delete    오른쪽 스와이프 = Keep →
```

### 앱-웹 연동 구조

```
[Share Extension]
  1. SavedItem 로컬 저장 (즉시)
  2. POST /api/v1/share  →  response.id = serverContentId
  3. StorageService.updateItemById(item with serverContentId)

[ItemDetailView: Keep/Delete]
  1. StorageService.updateItemById(status: .kept/.deleted)
  2. POST /api/v1/swipe { content_id: serverContentId, action: "keep"/"discard" }
     → 웹 대시보드에서 Archived/Deleted 탭에 반영
```

### 탭 필터 매핑

| 탭 | 표시 조건 | 아이콘 (empty state) |
|----|-----------|----------------------|
| Inbox | `status == .unread` | `tray` |
| Archived | `status == .kept` (+ 기존 `.read`, `.discarded`) | `archivebox` |
| Deleted | `status == .deleted` | `trash` |

## 4. Data Models

### 기존 모델 변경

**`SavedItem.Status`** (SavedItem.swift)

```swift
// 변경 전
enum Status: String, Codable {
    case unread, read, discarded
}

// 변경 후
enum Status: String, Codable {
    case unread     // Inbox (기본값, 저장 직후)
    case read       // 기존 호환용 — Archived로 취급
    case discarded  // 기존 호환용 — Archived로 취급
    case kept       // 사용자가 Keep 선택 → Archived 탭
    case deleted    // 사용자가 Delete 선택 → Deleted 탭
}
```

> `read` / `discarded`는 기존 저장 데이터 호환을 위해 유지. LibraryFilter에서 Archived 탭에 포함시킨다.

### Existing Used

- `SavedItem` — status 필드에 `.kept` / `.deleted` 케이스 추가
- `StorageService.updateItem()` — 상태 변경 후 로컬 저장 (기존 메서드 그대로 사용)

## 5. API Design

백엔드에 Keep/Delete를 위한 Swipe API가 이미 존재한다. 웹 대시보드도 동일 API를 사용한다.

### 주요 엔드포인트

| Endpoint | Method | 설명 |
|----------|--------|------|
| `/api/v1/swipe` | POST | Keep 또는 Discard 단건 기록 |
| `/api/v1/swipe/batch` | POST | Keep/Discard 배치 기록 |
| `/api/v1/content/pending` | GET | Inbox 목록 (아직 swipe하지 않은 콘텐츠) |
| `/api/v1/content/kept` | GET | Archived 목록 (Keep 처리된 콘텐츠) |
| `/api/v1/content/discarded` | GET | Deleted 목록 (Discard 처리된 콘텐츠) |
| `/api/v1/content/{content_id}/status` | PATCH | 상태 직접 변경 (INBOX → ARCHIVED) |
| `/api/v1/content/{content_id}` | DELETE | 소프트 삭제 (Trash 이동, 30일 복원 가능) |
| `/api/v1/content/{content_id}/restore` | POST | Trash에서 복원 |

### Keep / Delete 핵심 흐름 — `POST /api/v1/swipe`

**Request:**
```json
{
  "content_id": 42,
  "action": "keep"
}
```
> `action` 값: `"keep"` (Archived로) 또는 `"discard"` (Deleted로)

**Response (201 Created):**
```json
{
  "id": 101,
  "content_id": 42,
  "action": "keep"
}
```

### 서버 상태 ↔ 앱 탭 매핑

| iOS 탭 | 서버 API | action 값 |
|--------|----------|-----------|
| Inbox | `GET /content/pending` | — (swipe 없음) |
| Archived | `GET /content/kept` | `"keep"` |
| Deleted | `GET /content/discarded` | `"discard"` |

### 중요 제약: content_id (서버 정수 ID)

`POST /api/v1/swipe`는 서버에서 발급한 **정수형 content_id**를 요구한다.
현재 `SavedItem`은 로컬 `UUID`만 보유하고 있어, swipe API 호출을 위해 **서버 content_id를 `SavedItem`에 저장하는 필드 추가**가 필요하다.

```swift
// SavedItem에 추가 필요
var serverContentId: Int?   // POST /share 응답의 id 값
```

`POST /share` 응답의 `id` 필드를 저장하여 이후 swipe 호출 시 사용한다.

## 6. Implementation

### Files

- `Briefly/Models/SavedItem.swift` — `Status` enum에 `.kept`, `.deleted` 추가; `serverContentId: Int?` 필드 추가
- `Briefly/Services/BrieflyAPI.swift` — `swipe(contentId:action:token:)` 메서드 추가
- `Briefly/Views/LibraryView.swift` — `LibraryFilter` 3탭, `InboxNavigation` 타입 추가, navigationDestination 분기
- `Briefly/Views/ItemDetailView.swift` — 카드 리더 모드: items 배열 + currentIndex, 하단 고정 액션 바, 수평 스와이프 제스처, 자동 어드밴스

### Key Logic

1. **SavedItem 확장**
   ```swift
   // Status에 추가
   case kept     // Keep → Archived
   case deleted  // Delete → Deleted

   // 새 필드 (서버 swipe API용)
   var serverContentId: Int?
   ```

2. **LibraryView: InboxNavigation 타입 + 3탭**
   ```swift
   struct InboxNavigation: Hashable {
       let items: [SavedItem]   // 탭한 시점의 Inbox 스냅샷
       let startIndex: Int
   }

   enum LibraryFilter: String, CaseIterable {
       case inbox = "Inbox", archived = "Archived", deleted = "Deleted"
   }

   // Inbox 탭: InboxNavigation으로 이동
   // Archived/Deleted 탭: SavedItem으로 이동
   ```

3. **ItemDetailView: 카드 리더 모드**
   ```swift
   struct ItemDetailView: View {
       let items: [SavedItem]      // Inbox 전체 목록 or [singleItem]
       let showActions: Bool       // Inbox 진입 시만 true
       @State var currentIndex: Int
       @State var dragOffset: CGFloat = 0
       @Environment(\.dismiss) var dismiss

       var currentItem: SavedItem { items[currentIndex] }
   }
   ```

4. **수평 스와이프 제스처 (ScrollView와 공존)**
   ```swift
   ScrollView(.vertical) { ... }
       .simultaneousGesture(
           DragGesture(minimumDistance: 20)
               .onChanged { v in
                   guard abs(v.translation.width) > abs(v.translation.height) else { return }
                   dragOffset = v.translation.width
               }
               .onEnded { v in
                   if v.translation.width > 100 { performKeep() }
                   else if v.translation.width < -100 { performDelete() }
                   else { withAnimation(.spring()) { dragOffset = 0 } }
               }
       )
       .offset(x: dragOffset)   // 스와이프 중 시각 피드백
   ```

5. **카드 전환 애니메이션**
   - Keep: `dragOffset = +screenWidth * 1.5` (0.25s) → reset → `currentIndex += 1`
   - Delete: `dragOffset = -screenWidth * 1.5` (0.25s) → reset → `currentIndex += 1`
   - 마지막 카드 처리 후: `dismiss()`
   - 전환 시 `NotificationCenter.post(.fetchCoordinatorDidUpdate)` → LibraryView 자동 reload

6. **서버 API 연동 (swipe)**
   ```swift
   // performKeep / performDelete 내부
   if let contentId = currentItem.serverContentId,
      let token = AuthTokenStore.shared.accessToken {
       Task { try? await BrieflyAPI.shared.swipe(contentId: contentId, action: action, token: token) }
   }
   ```

7. **Floating Filter Tab 너비 조정**
   - 3개 탭: padding `.horizontal, 14` 로 축소

### Dependencies

**Requires**:
- `IOS-001` — StorageService, SavedItem 기반 구조, BrieflyAPI, AuthTokenStore
- 백엔드 `POST /api/v1/swipe` — 이미 구현·배포 완료 (웹 대시보드 동일 사용)

**Provides**:
- `UX-008` (향후) — Deleted 탭에서 완전 삭제(purge) 기능

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Inbox 마지막 카드 처리 | `currentIndex + 1 == items.count` → `dismiss()` 호출, Library로 복귀 |
| Inbox가 1개일 때 진입 | 카드 1개만 처리하고 dismiss |
| Archived / Deleted 상세 진입 | `showActions = false` → 액션 바 없음, 스와이프 비활성화 |
| 기존 `.read` / `.discarded` 상태 데이터 | Archived 탭 필터 조건에 포함시켜 정상 표시 |
| Keep/Delete 연타 | 애니메이션 중 `isProcessing` flag로 버튼/제스처 비활성화 |
| 스와이프 미완성 (threshold 미만) | `withAnimation(.spring()) { dragOffset = 0 }` 으로 복귀 |
| `serverContentId` 없음 (오프라인 저장) | 로컬 상태 변경만 수행, API 호출 skip (fire-and-forget) |
| 네트워크 오류 (swipe API 실패) | 로컬은 이미 처리됨 → 서버 오류 무시 (백그라운드 재시도는 향후) |
| 저장소 접근 실패 | `StorageService.updateItem` 내 기존 guard 처리 유지 |

## 8. Testing

- **수동 검증 (Inbox → Keep)**:
  1. Share Extension으로 링크 저장 → Library > Inbox에 표시 확인
  2. 카드 탭 → 상세 페이지 하단 Keep 버튼 탭
  3. 상세 닫힘 → Inbox에서 사라짐 확인
  4. Archived 탭에서 해당 아이템 표시 확인

- **수동 검증 (Inbox → Delete)**:
  1. 동일 흐름으로 Delete 버튼 탭
  2. Deleted 탭에서 표시 확인

- **하위 호환성**:
  - 기존 `.read` / `.discarded` 상태 아이템이 Archived 탭에 정상 표시되는지 확인

- **Acceptance**:
  - 3개 탭이 정상 전환됨
  - Keep/Delete 버튼이 Inbox 상세에서만 노출됨
  - 결정 후 자동 pop 동작

## 9. Sensory Verification

- **Visual (시각)**: Library 탭에 Inbox / Archived / Deleted 3개 탭이 캡슐 형태로 표시됨. 카드 리더 모드 하단에 Delete(빨간 텍스트) / Skip Now(회색 텍스트) / Keep(파란 텍스트) 버튼이 구분선으로 나뉘어 고정 표시. 스와이프 중 카드 상단에 DELETE(빨간)/KEEP(초록) 배지 인디케이터 표시. 카드가 해당 방향으로 실시간 이동하고, 처리 시 날아가며 다음 카드로 전환.
- **Auditory (청각)**: 상태 변경 시 `NotificationCenter` 이벤트 → `viewModel.reload()`. (선택) 카드 전환 시 haptic feedback.
- **Tactile (촉각)**: 스와이프 드래그 중 실시간 카드 offset 이동. threshold 미달 시 spring 애니메이션으로 복귀. 버튼 탭 → 0.25s 슬라이드 아웃 → 즉시 다음 카드.

## 10. Future Enhancements

1. **Undo 기능** — Keep/Delete 직후 토스트 알림 + "취소" 버튼으로 결정 되돌리기
2. **Deleted 탭 완전 삭제** — Deleted 탭에서 선택 삭제 또는 전체 비우기 (`DELETE /api/v1/content/{id}/purge`)
3. **서버 → 앱 동기화** — 앱 실행 시 `GET /api/v1/content/pending` 등으로 서버 상태를 로컬에 머지
4. **스와이프 인디케이터** — 스와이프 중 Keep/Delete 레이블 오버레이 표시 (Tinder-style)
5. **Archived → Inbox 복원** — Archived 상세에서 "Inbox로 되돌리기" 버튼 (`POST /api/v1/content/{id}/restore`)
