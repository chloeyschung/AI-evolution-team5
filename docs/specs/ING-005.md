# ING-005: iOS page_text 전송으로 SPA/JS 렌더링 사이트 AI 요약 지원

**Status**: Planned | **Created**: 2026-06-04 | **Author**: suyoung
**F-xxx Mapping**: F-005 (AI Summary) | **Phase**: Phase 3 - Enhancement | **Priority**: High

---

## 1. Overview

**Problem**: yozm.wishket.com처럼 JavaScript로 렌더링되는 SPA 사이트는 백엔드 서버 사이드 HTTP 스크래핑으로 실제 기사 내용을 가져올 수 없다. 결과적으로 AI 요약이 사이트 전체 설명(플랫폼 홍보 문구)으로 생성되거나, 아예 요약이 없는 상태가 된다.

**Root Cause**: 현재 흐름의 구조적 한계:
1. iOS Share Extension: URL만 백엔드에 전송 (`SharePayload` = `content` + `platform`)
2. 백엔드 `_background_summarize`: 서버 사이드 HTTP로 URL 콘텐츠 수신 → SPA는 JS 없이 빈 HTML
3. iOS FetchCoordinator: 메인 앱에서 WKWebView로 실제 본문 취득 성공 → 로컬 표시만, 백엔드 미전송

**Solution**: FetchCoordinator가 WKWebView로 기사 본문 취득 성공 시, 해당 텍스트를 백엔드에 전송해 AI 요약을 트리거한다. 백엔드에는 `page_text`를 받아 요약을 (재)실행하는 엔드포인트를 추가한다.

---

## 2. 문제 진단

### 현재 흐름

```
[Share Extension]
URL 공유 → appendToInbox(SavedItem) → BrieflyAPI.share(url:token:)
→ POST /share { content: url, platform: host }
→ 백엔드: _background_summarize(url) → fetch_html_and_text(url)
→ SPA: JS 없이 <meta> 태그만 → 플랫폼 설명 → AI 요약 = 플랫폼 전체 설명

[Main App — FetchCoordinator]
drainInboxAndLoad() → fetchIfNeeded(items)
→ fetchOne(item): MetadataService(OG) + fetchArticleText(WKWebView)
→ WKWebView 성공: articleText = 실제 기사 본문 (수천 자)
→ StorageService.updateItem(updated)  ← 로컬에만 저장, 백엔드 미전송
```

### 갭

iOS는 WKWebView로 실제 본문을 가지고 있지만, 백엔드는 이를 알지 못해 빈/나쁜 데이터로 요약한다.

## 3. 수정 방안

### 백엔드: 재요약 엔드포인트 추가

`POST /api/v1/content/{id}/rescan`

- `page_text`를 body로 받아 `_background_summarize`를 재실행
- 이미 요약이 있더라도 `page_text`가 제공된 경우 재요약 허용 (클라이언트가 더 좋은 텍스트를 가진 경우)
- `is_ai_summarized = True`인 항목도 재요약 가능 (`force` 플래그로 제어)

### iOS: FetchCoordinator에서 page_text 백엔드 전송

`fetchOne(item:)` 완료 후:
- `item.serverContentId != nil` (서버에 등록된 항목)
- `articleText`가 충분한 길이 (`>= 200자`)
- `item.summary == nil` (아직 요약 없음) — 혹은 요약이 있어도 기존 요약이 짧거나 플랫폼 설명인 경우

→ `BrieflyAPI.shared.rescan(contentId:pageText:token:)` 호출

## 4. 구현 상세

### 4-1. 백엔드: `POST /api/v1/content/{id}/rescan`

**파일**: `src/api/routers/content.py`

```python
class RescanRequest(BaseModel):
    page_text: str = Field(..., min_length=50, description="Client-extracted page text")
    force: bool = Field(False, description="Re-summarize even if already summarized")


@router.post("/content/{content_id}/rescan", status_code=202)
async def rescan_content(
    content_id: int,
    data: RescanRequest,
    background_tasks: BackgroundTasks,
    user_id: int = Depends(get_current_user),
    db: AsyncSession = Depends(get_db),
) -> dict:
    repo = ContentRepository(db)
    content = await repo.get_one(content_id)
    if not content or content.user_id != user_id or content.is_deleted:
        raise HTTPException(status_code=404)

    already = bool(getattr(content, "is_ai_summarized", False))
    if already and not data.force:
        return {"status": "skipped", "reason": "already_summarized"}

    share_handler = get_share_handler()
    background_tasks.add_task(
        _background_summarize,
        content_id,
        user_id,
        content.url,
        share_handler._content_extractor,
        share_handler._summarizer,
        data.page_text,
    )
    return {"status": "queued"}
```

**참고**: `_background_summarize`는 `page_text`가 있으면 서버 사이드 fetch를 건너뛰고 그대로 사용 (`if page_text: text_content = page_text`). 백엔드 변경 없이 기존 로직 재사용.

`already_summarized` 로직 수정 — `page_text` rescan 시 기존 요약을 덮어쓸 수 있도록:

```python
# _background_summarize 내부
already_summarized = (
    bool(getattr(content, "is_ai_summarized", False)) or bool(content.summary)
) and not page_text   # ← page_text가 있으면 재요약 허용
```

**파일**: `src/api/schemas.py`

```python
class RescanRequest(BaseModel):
    page_text: str = Field(..., min_length=50)
    force: bool = False
```

### 4-2. iOS: `BrieflyAPI.rescan`

**파일**: `Briefly/Briefly/Services/BrieflyAPI.swift`

```swift
struct RescanPayload: Encodable {
    let pageText: String
    let force: Bool

    enum CodingKeys: String, CodingKey {
        case pageText = "page_text"
        case force
    }
}

func rescan(contentId: Int, pageText: String, token: String) async throws {
    let body = RescanPayload(pageText: pageText, force: false)
    let _: EmptyResponse = try await post("/content/\(contentId)/rescan", body: body, token: token)
}
```

### 4-3. iOS: FetchCoordinator에서 rescan 호출

**파일**: `Briefly/Briefly/Services/FetchCoordinator.swift`

`fetchOne(item:)` 2단계 완료 후:

```swift
// 2단계 완료 후 page_text 백엔드 전송
let articleText = updated.articleText ?? ""
let hasSufficientText = articleText.count >= 200
let needsSummary = item.summary == nil || item.summary?.isEmpty == true
let hasServerId = item.serverContentId != nil

if hasSufficientText && needsSummary, hasServerId,
   let contentId = item.serverContentId,
   let token = AuthTokenStore.shared.accessToken {
    Task {
        try? await BrieflyAPI.shared.rescan(contentId: contentId, pageText: articleText, token: token)
        print("[Fetch] page_text 백엔드 전송 완료: contentId=\(contentId), \(articleText.count)자")
    }
}
```

## 5. 테스트 시나리오

| 시나리오 | 기대 결과 |
|---------|----------|
| yozm.wishket.com 기사 공유 | FetchCoordinator WKWebView 성공 → rescan 전송 → 기사 본문 기반 AI 요약 |
| 일반 서버 렌더링 사이트 | 백엔드 기존 요약 이미 정확 → `needsSummary = false` → rescan 미호출 |
| YouTube | `isYouTubeURL` → step 2 WKWebView 없음 → rescan 미호출 (transcript 경로 사용) |
| LinkedIn oEmbed 성공 | `articleText = ogDescription` (짧음, <200자 가능) → rescan 호출 여부는 길이 기준 |
| 오프라인 / 토큰 없음 | `try?` 로 무시, 로컬 표시는 정상 유지 |
| WKWebView 실패 (`articleText = nil`) | `hasSufficientText = false` → rescan 미호출 |

## 6. 보안 / 부작용 고려

- **인증**: `Depends(get_current_user)` — 본인 콘텐츠만 rescan 가능
- **소유권 검증**: `content.user_id != user_id` 체크로 타인 콘텐츠 rescan 차단
- **rate limit**: 기존 `/share` rate limit(120/min)과 별도, 필요 시 `@limiter.limit("30/minute")` 추가
- **텍스트 크기**: `min_length=50` + `_background_summarize` 내 `text[:6000]` 슬라이싱으로 과도한 입력 처리
- **중복 호출**: `needsSummary` 조건으로 이미 요약 있는 항목은 rescan 미호출

## 7. 한계

- **Share Extension 시점**: 공유 직후 바로 요약이 생성되지 않음. FetchCoordinator가 메인 앱에서 실행 완료 후 전송이므로, 앱 열기 전까지 요약 없음 (기존과 동일)
- **WKWebView 미지원 사이트**: JS 없이도 콘텐츠가 없는 경우 iOS WKWebView도 실패 → rescan 미호출 → 요약 없음
- **기존 잘못된 요약 유지**: `needsSummary = item.summary == nil`이므로, 이미 나쁜 요약(플랫폼 설명)이 저장된 기존 항목은 자동 재요약 안됨. 수동으로 `force=true` rescan 필요 (향후 UI에서 지원 가능)
