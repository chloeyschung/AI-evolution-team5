# DAT-003: Soft Delete & Data Recovery

**Status**: Design | **Created**: 2026-04-17 | **Author**: abraxaspark
**F-xxx Mapping**: N/A (Infrastructure) | **Phase**: Cross-Cutting | **Priority**: High

---

## 1. Overview

**Problem**: Hard deletes (`DELETE FROM ...`) are irreversible. Users who accidentally discard content or delete their account have no recovery path, and the system loses audit evidence for debugging and support.

**Solution**: Replace hard deletes with a soft-delete pattern: add an `is_deleted` boolean column to affected models and filter it in all read paths. A time-bounded recovery window (e.g. 30 days) allows restoration before permanent purge.

**Goals**: Reversible deletions for content and accounts, preserved swipe history for analytics, no data loss during accidental actions.

**Non-Goals**: Full version history / content diffing, admin-level un-delete UI (Phase 3+), cross-device undo (requires real-time sync).

## 3. User Story / Behavior

As a user, I want my "deleted" content to be recoverable for 30 days, so that accidental swipe-offs don't cause permanent data loss.

### Key Behaviors

- Deleting content sets `is_deleted = True` and `deleted_at = utc_now()`; the row stays in the DB
- All queries that list or fetch content add `WHERE is_deleted = False` automatically
- Within 30 days, a restore endpoint sets `is_deleted = False` and clears `deleted_at`
- After 30 days, a scheduled background task permanently purges rows where `deleted_at < utc_now() - 30d`
- Account deletion soft-deletes `UserProfile` and cascades to soft-delete all owned `Content` and `InterestTag` rows

## 4. Data Models

### Schema Changes

**Content** — add columns:
```
is_deleted: bool = False   (indexed)
deleted_at: datetime | None = None
```

**UserProfile** — add columns:
```
is_deleted: bool = False   (indexed)
deleted_at: datetime | None = None
```

**InterestTag** — add columns:
```
is_deleted: bool = False
deleted_at: datetime | None = None
```

**SwipeHistory** — no change (rows are never deleted).

### Migration

These are additive columns with defaults; no data migration needed. Existing rows get `is_deleted = False` by default at migration time.

## 5. API Design

| Endpoint | Method | Description |
|----------|--------|-------------|
| `/api/v1/content/{id}` | DELETE | Soft-delete content (sets `is_deleted`, does not remove row) |
| `/api/v1/content/{id}/restore` | POST | Restore soft-deleted content within recovery window |
| `/api/v1/auth/account/delete` | POST | Soft-delete account + cascade to owned content |

### Response Examples

**DELETE /api/v1/content/{id}** (soft delete)

Response (200 OK):
```json
{
  "id": 42,
  "is_deleted": true,
  "deleted_at": "2026-04-17T10:00:00Z",
  "recoverable_until": "2026-05-17T10:00:00Z"
}
```

**POST /api/v1/content/{id}/restore**

Response (200 OK):
```json
{
  "id": 42,
  "is_deleted": false,
  "deleted_at": null
}
```

Error (410 Gone) if recovery window has passed:
```json
{ "detail": "recovery_window_expired" }
```

## 6. Implementation

### Files

- `src/data/models.py` — add `is_deleted`, `deleted_at` to `Content`, `UserProfile`, `InterestTag`
- `src/data/repository.py` — update all `select()` queries to filter `is_deleted == False`; add `restore_content()`; soft-delete cascade in `delete_user()`
- `src/api/routes.py` — change `delete_content` and `delete_account` route handlers to call soft-delete; add restore endpoint
- `src/api/schemas.py` — add `DeletedContentResponse` with `recoverable_until`

### Key Logic

1. **Soft-delete helper**: `_soft_delete(obj)` sets `obj.is_deleted = True`, `obj.deleted_at = utc_now()` — reused across models.

2. **Query filter**: All `ContentRepository` list/get methods add `.where(Content.is_deleted == False)`. Apply at the repository layer, not service layer, so callers never have to think about it.

3. **Account cascade**: Deleting a `UserProfile` calls `_soft_delete()` on all owned `Content` and `InterestTag` rows via a bulk `UPDATE ... SET is_deleted=True WHERE user_id=?`.

4. **Purge task**: Background job (APScheduler or equivalent) runs daily: `DELETE FROM content WHERE is_deleted = True AND deleted_at < utc_now() - 30d`.

### Dependencies

**Requires**: DAT-001 (Content model), DAT-002 (UserProfile model)

**Provides**: safe account deletion for AUTH-005 (two-step deletion flow)

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Restore after recovery window | 410 Gone — `recovery_window_expired` |
| Restore content belonging to deleted account | 404 — account no longer active |
| Swipe on soft-deleted content | 404 — excluded from all queries |
| Duplicate soft-delete call | Idempotent — already `is_deleted = True`, return 200 |
| Purge task races with restore | Restore checks `deleted_at < cutoff`; if purge already ran, row is gone → 404 |

## 8. Testing

- **Unit**: `_soft_delete()` helper, cascade logic, recovery window boundary (29d OK, 31d 410)
- **Integration**: Full delete → restore flow, account cascade, purge task dry-run
- **Acceptance**: Deleted content invisible in list; restore brings it back; permanent purge cleans rows older than 30d

## 9. Sensory Verification

- **Visual**: Deleted content does not appear in inbox or archive lists
- **Auditory**: Restore endpoint returns 200 with `is_deleted: false`; expired restore returns 410
- **Tactile**: Bulk soft-delete (account cascade) completes < 200ms for up to 10k rows
