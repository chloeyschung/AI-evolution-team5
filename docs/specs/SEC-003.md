# SEC-003: Audit Logging & Security Event Tracking

**Status**: Design | **Created**: 2026-04-17 | **Author**: abraxaspark
**F-xxx Mapping**: N/A (Infrastructure) | **Phase**: Cross-Cutting | **Priority**: High

---

## 1. Overview

**Problem**: Security-sensitive actions (login, token refresh, account deletion) leave no tamper-evident trail. Incidents cannot be reconstructed, anomalous behavior cannot be detected, and compliance requirements for access logs cannot be met.

**Solution**: Introduce an `AuditLog` table that records security events with structured metadata. Routes emit audit events inline (same DB transaction as the action) so logs are never out of sync with actual state changes.

**Goals**: Durable per-user event trail, structured enough for programmatic queries, low overhead (single insert per event).

**Non-Goals**: Real-time alerting or anomaly detection (Phase 4+), log export / SIEM integration, audit log UI for end users.

| NFR-3 | `user_id` + `created_at` composite index for per-user event queries | P0 |

## 3. User Story / Behavior

As a system operator, I want a tamper-evident log of authentication events, so that I can reconstruct what happened during a security incident without relying on application logs alone.

### Key Behaviors

- Every successful login appends one `AuditLog` row with `event_type=LOGIN_SUCCESS`
- Every failed login attempt appends one row with `event_type=LOGIN_FAILURE` and reason in `metadata`
- Token refresh appends `REFRESH_TOKEN` event
- Account delete step 1 appends `ACCOUNT_DELETE_INITIATED`; step 2 appends `ACCOUNT_DELETE_CONFIRMED`
- All inserts happen in the same DB session as the action â€” if the action rolls back, the log row rolls back too

## 4. Data Models

### New Table: `audit_logs`

```
AuditLog:
  id:           Integer, PK, auto-increment
  user_id:      Integer, nullable (NULL for pre-auth failures), indexed
  event_type:   String(50), NOT NULL, indexed   (see enum below)
  ip_address:   String(45), nullable             (IPv4 or IPv6)
  metadata:     JSON, nullable                   (event-specific detail)
  created_at:   DateTime, NOT NULL, default=utc_now, indexed

Composite index: (user_id, created_at DESC)
```

### Event Type Enum: `AuditEventType`

```python
class AuditEventType(str, Enum):
    LOGIN_SUCCESS           = "login_success"
    LOGIN_FAILURE           = "login_failure"
    LOGOUT                  = "logout"
    REFRESH_TOKEN           = "refresh_token"
    ACCOUNT_DELETE_INITIATED = "account_delete_initiated"
    ACCOUNT_DELETE_CONFIRMED = "account_delete_confirmed"
    OAUTH_CONNECT           = "oauth_connect"
    OAUTH_DISCONNECT        = "oauth_disconnect"
    PASSWORD_RESET_REQUESTED = "password_reset_requested"
    PASSWORD_RESET_COMPLETED = "password_reset_completed"
```

### Metadata Examples

```json
// LOGIN_FAILURE
{ "reason": "invalid_password", "email": "user@example.com" }

// OAUTH_CONNECT
{ "provider": "youtube" }

// ACCOUNT_DELETE_INITIATED
{ "step": 1, "confirmation_token_issued": true }
```

## 6. Implementation

### Files

- `src/data/models.py` â€” add `AuditLog` model and `AuditEventType` enum
- `src/data/repository.py` â€” add `AuditRepository.log_event(user_id, event_type, ip_address, metadata)`
- `src/api/routes.py` â€” emit audit events at: login (lines ~1139), token refresh (~796), account delete (~1279, ~1362)
- `src/api/schemas.py` â€” no public schema needed (internal write-only)

### Key Logic

1. **`AuditRepository.log_event()`**: single insert, always called with the active session so it participates in the surrounding transaction. No separate commit.

2. **IP extraction**: Use FastAPI `Request.client.host`. Pass `Request` into routes that need it (login, delete). Routes that already receive `Request` (e.g. for rate limiting) already have access.

3. **Failure logging**: On caught `HTTPException` for auth failures, still log before re-raising â€” use a nested try/except so the audit insert doesn't suppress the original error.

4. **No log-on-log**: `AuditRepository` itself does not log. Circular logging is prevented by design (no audit event for audit inserts).

### Dependencies

**Requires**: DAT-001 (DB session pattern), AUTH-005 (login/token refresh routes to instrument)

**Provides**: forensic trail for SEC-001 (security hardening) incident response

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Unauthenticated login failure (no user_id yet) | `user_id = NULL` in log row |
| DB failure on audit insert | Transaction rolls back including the triggering action â€” consistent state |
| High-volume token refresh (mobile background sync) | Audit rows accumulate; purge policy (NFR-2) applies |
| IP behind proxy | Use `X-Forwarded-For` header if `TRUST_PROXY=true` in config; otherwise `client.host` |
