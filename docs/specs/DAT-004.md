# DAT-004: Cursor Pagination for iOS Feed Stability

**Status**: Design | **Created**: 2026-04-18 | **Author**: codex
**F-xxx Mapping**: N/A (API Contract Hardening) | **Phase**: Phase 1 - MVP (Mobile Focus) | **Priority**: High

---

## 1. Overview

**Problem**: Offset-based pagination (`offset`, `next_offset`) can duplicate or skip items when new content is inserted between page requests in mobile infinite-scroll flows.

**Solution**: Add cursor-based pagination (`cursor`, `next_cursor`) on core content-list endpoints using stable sort keys, while keeping offset fields for backward compatibility.

**Goals**: Stable forward-only pagination under concurrent inserts, additive API change with no immediate breakage for existing clients.

**Non-Goals**: Full offset deprecation in this phase, bidirectional cursors, cross-endpoint universal cursor format for all integrations routes.

## 3. User Story / Behavior

As an iOS user scrolling my feed, I want each next-page request to continue from the exact last item I saw, so that I never see random duplicates when new items are added.

### Key Behaviors

- First page can be requested without `cursor`
- `next_cursor` from page N is used as `cursor` for page N+1
- If `cursor` is present, server pagination position is derived from cursor and not from offset
- Invalid/expired/malformed cursor returns 400 with clear error code

## 4. Data Models

### Existing Used

- `Content` (`id`, `created_at`, `user_id`, `status`, `is_deleted`)
- `SwipeHistory` (for `kept` / `discarded` ordering)
- `ContentTag` (for `/search` and tag filters)

### Cursor Envelope

Opaque token encoding of the last row sort key (timestamp + tie-breaker ID) in URL-safe base64.

```json
  "items": [...],

}
```

## 6. Implementation

### Files

- `src/utils/cursor_pagination.py` - cursor encode/decode helpers
- `src/data/repository.py` - cursor-aware queries for content lists and search
- `src/api/routers/content.py` - cursor query arg + response `next_cursor`
- `src/api/routers/swipe.py` - cursor query arg + response `next_cursor`
- `src/api/schemas.py` - add `next_cursor` to paginated response schema

### Key Logic

1. Decode cursor token into `(timestamp, id)` sort boundary.
2. Query with strict `<` boundary on `(created_at, id)` equivalent ordering.
3. Fetch `limit + 1` rows to compute `has_more` and next cursor safely.
4. Keep offset flow untouched if `cursor` is not provided.

### Dependencies

**Requires**:
- DAT-001 (existing pagination wrapper)

**Provides**:
- iOS infinite-scroll contract stability

## 7. Edge Cases

| Scenario | Handling |
|----------|----------|
| Malformed cursor | 400 `invalid_cursor` |
| Cursor points past end | return empty list, `has_more=false`, `next_cursor=null` |
| Same timestamp rows | tie-break with `id` for deterministic page boundaries |
| Cursor + offset both provided | cursor path takes precedence (documented) |

## 9. Sensory Verification
